Active Directory

Servicio de directorio en una red distribuida

none

Unir un cliente Linux a Active Directory

Cliente Linux:

1. Si el dominio de la red de Active Directory tiene como TLD .local, es necesario editar el siguiente archivo:

nano /etc/nsswitch.conf

Dentro del archivo eliminaremos la sección mdns4_minimal [NOTFOUND=return], dejando la línea que lo tenía tal que así:

hosts: files dns

Esto se debe realizar porque .local se intenta resolver por mDNS, y eso impide que Linux consulte al DNS de Active Directory.


2. Hay que poner como DNS la IP del servidor de Active Directory. Verificación:

cat /etc/resolv.conf
ping <Subdominio_Servidor>.<Nombre_Dominio>.<Dominio_TLD>

No va a funcionar hacer ping sin especificar en dominio ya que no hay ningún dominio de búsqueda configurado para añadir ese dominio automáticamente a la consulta.


3. Ahora configuraremos el dominio de búsqueda, que permite contactar a un subdominio sin especificar el dominio completo, mediante el siguiente comando:

nm-connection-editor

Esto abrirá un gestor avanzado para configurar la red. Tendremos que ir a "Ajustes de IPv4" y añadir search <Nombre_Dominio>.<Dominio_TLD> en la sección de "Dominios de búsqueda".

Para verificarlo realizaremos una consulta ICMP a un subdominio sin especificar el dominio completo:

ping <Subdominio_Servidor>

También se podría hacer añadiendo la línea search <Nombre_Dominio>.<Dominio_TLD> en el archivo /etc/resolv.conf, pero al reiniciar el sistema no se aplicaria el cambio ya que el servicio Networking regenera su configuración en cada inicio del sistema.


4. Tenemos que asignar un hostname al cliente definiendo su dominio completo:

hostnamectl set-hostname <Subdominio_Cliente_Linux>.<Nombre_Dominio>.<Dominio_TLD>

Luego reinicie la terminal.

Para verificar los cambios se podría mirar que el siguiente archivo contenga el hostname actualizado:

cat /etc/hostname

5. Ahora estableceremos un registro DNS interno para el propio cliente:

nano /etc/hosts

Primero eliminaremos los registros que contengan el hostname antiguo del cliente:

127.0.0.1     <Hostname_Cliente_Antiguo>

Luego se añadirá un registro que apunte al propio cliente mediante su hostname y subdominio:

<IP_Cliente>     <Hostname_Cliente.dominio.tld>     <Hostname_Cliente>

6. Ahora instalaremos las dependencias necesarias:

apt update
apt install -y samba winbind krb5-user libnss-winbind libpam-winbind

Para verificar que se instalaron con éxito podremos consultar el siguiente archivo y verificar que default_realm deberia ser el nombre de dominio:

nano /etc/krb5.conf

7. A continuación tenemos que actualizar SAMBA mediante en siguiente archivo:

nano /etc/samba/smb.conf

Ahora nos dirigiremos a uns sección [GLOBAL] y cambiaremos las siguientes configuraciones:

7.1. Reemplazaremos el valor "WORKGROUP" de la siguiente línea, que es el grupo de trabajo:

workgroup = WORKGROUP

Por el nombre de dominio en mayúsculas y sin el TLD, por ejemplo como "CICLO" si el dominio fuera "ciclo.local":

workgroup = <dominio_sin_tld_y_mayusculas>

7.2. Debajo de "workgroup" añadiremos la siguiente línea, especificando el nombre de dominio completo:

realm = <nombre_dominio>.<dominio_tld>

7.3. Reemplazaremos el valor de la siguiente línea:

server role = standalone server

Por este otro valor, definiendo que el equipo actuará como miembro y no servidor:

server role = member server

7.4. A continuación tendremos que localizar un bloque de configuración de idmap en la sección [GLOBAL], que es parecido a lo siguiente:

;   idmap config * :              backend = tdb
;   idmap config * :              range   = 3000-7999
;   idmap config YOURDOMAINHERE : backend = tdb
;   idmap config YOURDOMAINHERE : range   = 100000-999999
;   template shell = /bin/bash

Lo reemplazaremos a la siguiente configuración:

    idmap config * :              backend = tdb
    idmap config * :              range   = 3000-7999
    template shell = /bin/bash
    template homedir = /home/%D/%u
    password server = *
    winbind use default domain = yes
    winbind enum users = yes
    winbind enum groups = yes

;   idmap config YOURDOMAINHERE : backend = tdb
;   idmap config YOURDOMAINHERE : range   = 100000-999999
  • idmap config * : backend = tdb: Esto indica que tdb es la base de datos.
  • idmap config * : range = 3000-7999: Es el rango de UIDs para los usuarios de Windows que se metan al dominio.
  • template shell = /bin/bash: La terminal por defecto que tendrá en usuario que se cree desde Windows.
  • template homedir = /home/%D/%u: La carpeta personal que tendrá en usuario que se cree desde Windows. Se crearán las carpetas con los nombres de usuario %u dentro una subcarpeta de /home con el nombre del dominio %D.
  • password server = *: Su utilizará automáticamente cualquier controlador de dominio disponible.
  • winbind use default domain = yes: Para evitar poner el nombre de usuario como CICLO\user o user@dominio al iniciar sesión.
  • winbind enum users = yes: Permite listar todos los usuarios del dominio.
  • winbind enum groups = yes: Permite enumerar los grupos del dominio.

7.5. Guardamos el archivo y salimos del editor de texto.


8. Ahora hay que unir el cliente al dominio:

8.1. Listaremos la información del dominio:

net ads info

8.2. Intentaremos unir el cliente al dominio mediante el usuario y contraseña del Administrador del servidor Windows:

net ads join -U Administrador

Si da un error en este paso es por hacer incorrectamente el paso 5 - /etc/hosts.

Si permite unirse al dominio, podremos ver el equipo en la sección "Computers" del servidor de Windows.

8.3. Listaremos la información del dominio:

net ads info

8.4. Reiniciaremos el servicio de winbind:

systemctl restart winbind.service

8.5. Listaremos la información de winbind, que debería listar los usuarios de Windows:

wbinfo -u

8.6. Deberiamos ser capaces de ver la información del usuario de Windows con el siguiente comando:

id <usuario_windows>

8.7. También deberiamos ser capaces de iniciar sesión con el usuario. Podemos probarlo con el siguiente comando:

login <usuario_windows>

Gracias al servicio libpam-winbind se puede iniciar sesión ya que modifica el fichero /etc/pam.d/common-auth, permitiendo comprobar las contraseñas tanto de windows como de Linux.

Tendremos el error de que no crea la carpeta personal, eso se soluciona editando /etc/pam.d/common-session - explicado en la guía sobre NIS: Problema - El cliente no tiene carpeta /home.

8.8. Por último tendríamos que mirar si la configuración de smb.conf se aplicó al usuario de Windows:

cat /etc/passwd | grep <usuario_windows>

Debe aparecer la configuración de UID, carpeta /home y shell que se configuró en samba.
