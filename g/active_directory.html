Crear un servidor Linux como controlador de dominio (DC)

Además de crear servidores Windows para que controlen Active Directory, podemos configurar un servidor Linux para que actúe como controlador de dominio sin necesidad de adquirir licencias y con las mismas funcionalidades.


1. Primero asignaremos al servidor una dirección IPv4 estática. Podemos mirar la dirección IP del sistema con:

ip a

2. Tenemos que asignar un hostname al cliente definiendo su dominio completo:

hostnamectl set-hostname <Hostname_Servidor_Linux>.<Nombre_Dominio>.<Dominio_TLD>

Para verificar los cambios se podría mirar que el siguiente archivo contenga el hostname actualizado:

cat /etc/hostname

3. Ahora estableceremos un registro DNS interno para cada equipo de la red:

nano /etc/hosts

Primero eliminaremos los registros que contengan el hostname antiguo del servidor:

127.0.0.1     <Hostname_Servidor_Linux_Antiguo>

Luego se añadirá un registro que apunte a cada IP de los equipos de la red mediante su hostname y dominio:

<IP_Servidor_Linux>     <Hostname_Servidor_Linux.dominio.tld>     <Hostname_Servidor_Linux>
<IP_Cliente_1>     <Hostname_Cliente_1.dominio.tld>     <Hostname_Cliente_1>
...

4. Ahora instalaremos los paquetes necesarios:

apt install -y samba winbind

5. Pararemos los servicios y deshabilitaremos:

systemctl stop samba-ad-dc smbd nmbd winbind
systemctl disable samba-ad-dc smbd nmbd winbind

6. Antes de configurar nada crearé una copia de seguridad:

mv /etc/samba/smb.conf /etc/samba/smb.conf.bak

7. Con el siguiente comando interactivo generaré una configuración de SAMBA:

samba-tool domain provision --use-rfc2307 --interactive

En "Realm" tiene que poner el nombre de dominio en mayúsculas.

En "Domain" tiene que poner el nombre de dominio sin el TLD y en mayúsculas.

En "Server Role" debe estar en dc (Domain Controller).

En "DNS Backend" se dejará como SAMBA_INTERNAL.

En "DNS Forwarders" tiene que definirse la IP de un servidor DNS local o de Internet, como 1.1.1.1.

Luego pedirá la contraseña de Administrador y empezará con el proceso de configuración.


8. Ahora verificaremos que la configuración sea la correcta:

cat /etc/samba/smb.conf

9. Es necesario editar el siguiente archivo para poder resolver correctamente consultas DNS:

nano /etc/nsswitch.conf

Dentro del archivo eliminaremos la sección mdns4_minimal [NOTFOUND=return], dejando la línea que lo tenía tal que así:

hosts: files dns myhostname

10. Por último iniciaremos el servicio winbind:

systemctl start samba-ad-dc smbd nmbd winbind
systemctl enable samba-ad-dc smbd nmbd winbind

Es importante que todos esos servicios estén activos para poder unirse al dominio. Si algo falla, se debe mirar el estado de cada servicio.


En la siguiente sección se puede ver la administración del servidor mediante línea de comandos.

Importante: El usuario administrador del dominio se llama administrator (en inglés).

Scripts de inicio de sesión desde un controlador de dominio (DC) Linux

Esta sección muestra como definir scripts de inicio de sesión para usuarios en equipos Windows. En equipos Linux es mejor usar PAM.

Para configurarlo, primero tendremos que editar el fichero de configuración de SAMBA:

nano /etc/samba/smb.conf

Añadiremos al final del archivo la siguiente sección, pero definiendo el nombre del dominio correctamente:

[netlogon]
    path = /var/lib/samba/sysvol/<dominio.tld>/scripts

Dentro de esa ruta se podrán guardar los scripts .bat que se ejecutarán en el cliente Windows.

La ruta al recurso compartido es el siguiente:

\\<Hostname_Servidor_Linux>\netlogon

Ahora se puede asociar el script al usuario mediante un equipo Windows con RSAT o con el siguiente comando en el servidor:

samba-tool user setprofile <Nombre_Usuario> --script-path <Nombre_Script.bat>

El script es una ruta relativa desde la ruta de sysvol configurada anteriormente.

En el equipo Windows con RSAT se puede configurar desde: WIN + R > dsa.msc > Usuario > Propiedades > Profile > Logon script.

Unir un cliente Linux a un controlador de dominio (DC)

Con los siguientes pasos en un cliente Linux podemos meter el equipo en un dominio creado por un controlador de dominio, tanto si es un servidor Linux o Windows:


1. Si el dominio de la red de Active Directory tiene como TLD .local, es necesario editar el siguiente archivo:

nano /etc/nsswitch.conf

Dentro del archivo eliminaremos la sección mdns4_minimal [NOTFOUND=return], dejando la línea que lo tenía tal que así:

hosts: files dns myhostname

Esto se debe realizar porque los dominios .local se intentan resolver por mDNS, y eso impide que Linux consulte al DNS de Active Directory.

Además, debemos añadir al final de las siguientes líneas la palabra winbind para permitir el inicio de sesión con usuarios del dominio:

passwd:         files systemd winbind
group:          files systemd winbind

2. Hay que poner como DNS la IP del servidor de Active Directory. Verificación:

cat /etc/resolv.conf
ping <Hostname_Servidor>.<Nombre_Dominio>.<Dominio_TLD>

No va a funcionar hacer ping sin especificar en dominio ya que no hay ningún dominio de búsqueda configurado para añadir ese dominio automáticamente a la consulta.


3. Ahora configuraremos el dominio de búsqueda, que permite contactar a un subdominio sin especificar el dominio completo, mediante el siguiente comando:

nm-connection-editor

Esto abrirá un gestor avanzado para configurar la red. Tendremos que ir a "Ajustes de IPv4" y añadir search <Nombre_Dominio>.<Dominio_TLD> en la sección de "Dominios de búsqueda".

Para verificarlo realizaremos una consulta ICMP a un subdominio sin especificar el dominio completo:

ping <Subdominio_Servidor>

También se podría hacer añadiendo la línea search <Nombre_Dominio>.<Dominio_TLD> en el archivo /etc/resolv.conf, pero al reiniciar el sistema no se aplicaría el cambio ya que el servicio Networking regenera su configuración en cada inicio del sistema.


4. Tenemos que asignar un hostname al cliente definiendo su dominio completo:

hostnamectl set-hostname <Hostname_Cliente_Linux>.<Nombre_Dominio>.<Dominio_TLD>

Para verificar los cambios se podría mirar que el siguiente archivo contenga el hostname actualizado:

cat /etc/hostname

5. Ahora estableceremos un registro DNS interno para el propio cliente:

nano /etc/hosts

Primero eliminaremos los registros que contengan el hostname antiguo del cliente:

127.0.0.1     <Hostname_Cliente_Antiguo>

Luego se añadirá un registro que apunte al propio cliente mediante su hostname y dominio:

<IP_Cliente>     <Hostname_Cliente.dominio.tld>     <Hostname_Cliente>

6. Ahora instalaremos los paquetes necesarios:

apt update
apt install -y samba winbind krb5-user libnss-winbind libpam-winbind

Para verificar que se instalaron con éxito podremos consultar el siguiente archivo y verificar que default_realm deberia ser el nombre de dominio en mayúsculas:

cat /etc/krb5.conf

7. A continuación tenemos que actualizar SAMBA mediante el siguiente archivo:

nano /etc/samba/smb.conf

Ahora nos dirigiremos a una sección [GLOBAL] y cambiaremos las siguientes configuraciones:

7.1. Reemplazaremos el valor "WORKGROUP" de la siguiente línea, que es el grupo de trabajo:

workgroup = WORKGROUP

Hay que poner el nombre de dominio en mayúsculas y sin el TLD, por ejemplo como "CICLO" si el dominio fuera "ciclo.local":

workgroup = <dominio_sin_tld_y_mayusculas>

7.2. Debajo de "workgroup" añadiremos la siguiente línea, especificando el nombre de dominio completo:

realm = <nombre_dominio>.<dominio_tld>

7.3. Reemplazaremos el valor de la siguiente línea:

server role = standalone server

Por este otro valor, definiendo que el equipo actuará como miembro y no servidor:

server role = member server

7.4. A continuación tendremos que localizar un bloque de configuración de idmap en la sección [GLOBAL], que es parecido a lo siguiente:

;   idmap config * :              backend = tdb
;   idmap config * :              range   = 3000-7999
;   idmap config YOURDOMAINHERE : backend = tdb
;   idmap config YOURDOMAINHERE : range   = 100000-999999
;   template shell = /bin/bash

Lo reemplazaremos a la siguiente configuración:

    idmap config * :              backend = tdb
    idmap config * :              range   = 3000-7999
    template shell = /bin/bash
    template homedir = /home/%D/%u
    password server = *
    winbind use default domain = yes
    winbind enum users = yes
    winbind enum groups = yes

;   idmap config YOURDOMAINHERE : backend = tdb
;   idmap config YOURDOMAINHERE : range   = 100000-999999
Explicación de la configuración
CONFIGURACIÓN EXPLICACIÓN 
idmap config * : backend = tdb
 Esto indica que tdb es la base de datos. 
idmap config * : range = 3000-7999
 Es el rango de UIDs para los usuarios de Windows que se metan al dominio. 
template shell = /bin/bash
 La terminal por defecto que tendrá en usuario que se cree desde Windows. 
template homedir = /home/%D/%u
 La carpeta personal que tendrá en usuario que se cree desde Windows. Se crearán las carpetas con los nombres de usuario %u dentro una subcarpeta de /home con el nombre del dominio %D. 
password server = *
 Se utilizará automáticamente cualquier controlador de dominio disponible. 
winbind use default domain = yes
 Para evitar poner el nombre de usuario como CICLO\user o user@dominio al iniciar sesión. 
winbind enum users = yes
 Permite listar todos los usuarios del dominio. 
winbind enum groups = yes
 Permite listar todos los grupos del dominio.

7.5. Guardamos el archivo y salimos del editor de texto.


8. Ahora hay que unir el cliente al dominio:

8.1. Listaremos la información del dominio:

net ads info

En caso de que de un error, hay que verificar que el paso anterior esté hecho correctamente.

8.2. Intentaremos unir el cliente al dominio mediante el usuario y contraseña del Administrador del servidor Windows:

net ads join -U Administrador

Si da un error en este paso es por hacer incorrectamente el paso 5 - /etc/hosts.

Si hay un error de DNS, es porque ya hubo otro cliente con el mismo nombre y dominio en el servidor.

Si permite unirse al dominio, podremos ver el equipo en la sección "Computers" del servidor de Windows.

8.3. Listaremos la información del dominio:

net ads info

8.4. Reiniciaremos el servicio de winbind:

systemctl restart winbind.service

8.5. Listaremos la información de winbind, que debería listar los usuarios de Windows:

wbinfo -u

8.6. Deberíamos ser capaces de ver la información del usuario de Windows con el siguiente comando:

id <usuario_windows>

8.7. También deberíamos ser capaces de iniciar sesión con el usuario. Podemos probarlo con el siguiente comando:

login <usuario_windows>

Gracias al servicio libpam-winbind se puede iniciar sesión ya que modifica el fichero /etc/pam.d/common-auth, permitiendo comprobar las contraseñas tanto de windows como de Linux.

Tendremos el error de que no crea la carpeta personal, se puede solucionar compartiendo la carpeta home del usuario al cliente o con el siguiente comando, que sirve para permitir a pam crear las carpetas en el cliente:

pam_auth_update

Hay que seleccionar la opción "Create home directory on login". Después podemos probar a iniciar sesión de nuevo y verificar que la carpeta se crea localmente.

8.8. Por último tendríamos que mirar si la configuración de smb.conf se aplicó al usuario de Windows:

cat /etc/passwd | grep <usuario_windows>

Debe aparecer la configuración de UID, carpeta /home y shell que se configuró en samba.

Montar en un cliente Linux una carpeta del controlador de dominio (DC)

1. Primero se instalarán los paquetes necesarios:

apt update
apt install -y libpam-mount cifs-utils

2. Ahora editaremos el archivo de configuración de pam_mount:

nano /etc/security/pam_mount.conf.xml

Ahora nos dirigiremos a la sección del archivo con el siguiente comentario en XML:

<!-- Volume definitions -->

Justo después de esa línea añadiremos nuevas etiquetas "volume" por cada carpeta a montar:

<volume
    fstype="cifs"
    server="servidor_win.adrian.local"
    path="nombre_recurso_compartido"
    mountpoint="~/ruta_montar_localmente"
    sgrp="nombre_grupo_permitido"
/>

Explicación de la configuración
CONFIGURACIÓN EXPLICACIÓN 
fstype=""
 Tipo de sistema de ficheros de la carpeta a montar. 
server=""
 Subdominio completo del servidor. 
path=""
 Nombre del recurso compartido del servidor. 
mountpoint=""
 La ruta local del cliente donde montar el recurso compartido del servidor. 
sgrp=""
 Nombre del grupo o grupos separados por comas cuyos miembros están permitidos para montar el recurso.

En caso de querer montar una carpeta cuyo nombre sea el del usuario que la monta podemos usar la variable %(USER), como por ejemplo:

<volume
    ...
    path="nombre_recurso_compartido/%(USER)"
    ...
/>

Hay otro tipo de variables que se pueden ver ejecutando el siguiente comando:

man pam_mount.conf

Unir un cliente Windows a un controlador de dominio (DC)

1. Primero configuraremos como DNS del cliente la IP del servidor / controlador de dominio.

2. En la sección "Cambios en el dominio o nombre de equipo" asignaremos un nombre de equipo identificativo y especificaremos el nombre de dominio del servidor.

Para unirse al dominio pedirá el usuario administrator y su contraseña asignada.

3. Reiniciaremos el equipo y luego tendríamos que ser capaces de poder iniciar sesión con un usuario del dominio.
