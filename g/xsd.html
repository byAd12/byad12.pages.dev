Ejemplo

<?xml version="1.0"?>
<xs:schema version="1.0" xmlns:xs="http://www.w3.org/2001/XMLSchema" elementFormDefault="qualified">
    <!-- El elemento raíz es "cartelera" -->
    <xs:element name="cartelera" type="tCartelera" />
  
    <!-- Dentro de "cartelera" va a haber hasta varios elementos "pelicula" -->
    <xs:complexType name="tCartelera">
        <xs:sequence>
            <xs:element name="pelicula" type="tPelicula" maxOccurs="unbounded" />
        </xs:sequence>
    </xs:complexType>

    <!-- El elemento "pelicula" tendrá una secuencia de sub-elementos dentro -->
    <xs:complexType name="tPelicula">
        <xs:sequence>
            <xs:element name="titulo" type="xs:string"  />
            <xs:element name="titulo-original" type="xs:string" minOccurs="0" maxOccurs="1" />
            <xs:element name="pais" type="xs:string" />
            <xs:element name="generos" type="tGenero" />
            <!-- Se define con xs:choice que solo pueda haber un elemento u otro -->
            <xs:choice>
                <xs:element name="clasificacion" type="tClasificacion" />
                <xs:element name="sin-clasificar" type="tSinClasificar" />
            </xs:choice>
            <xs:element name="sinopsis" type="xs:string" />
            <xs:choice>
                <xs:element name="director" type="xs:string" />
                <xs:element name="directora" type="xs:string" />
            </xs:choice>
            <xs:element name="reparto" type="tReparto"/>
            <xs:element name="web" type="url" minOccurs="0" maxOccurs="1"/>
            <xs:element name="imdb" type="url"/>
            <xs:element name="cartel" type="xs:string"/>
        </xs:sequence>
        <!-- Después de la secuencia se define qué atributos tendrá el elemento "pelicula" -->
        <xs:attribute name="codigo" type="xs:ID" use="required" />
        <xs:attribute name="duracion" type="xs:positiveInteger" use="required" />
        <xs:attribute name="ano" type="xs:positiveInteger" use="required" />
    </xs:complexType>

    <!-- Se define una secuencia de sub-elementos dentro de la etiqueta "generos" -->
    <xs:complexType name="tGenero">
        <xs:sequence>
            <xs:element name="genero" type="xs:string" maxOccurs="unbounded" />
        </xs:sequence>
    </xs:complexType>

    <!-- Se define que "clasificacion" tenga un atributo "edad" -->
    <xs:complexType name="tClasificacion">
        <xs:attribute name="edad" type="tEdad" use="required"/>
    </xs:complexType>

    <!-- En el atributo "edad" se define su tipo de valor y valor mínimo -->
    <xs:simpleType name="tEdad">
        <xs:restriction base="xs:integer">
            <xs:minInclusive value="0" />
        </xs:restriction>
    </xs:simpleType>

    <!-- El elemento "sin-clasificar" se define como vacío -->
    <xs:complexType name="tSinClasificar" />

    <!-- En el elemento "reparto" se indica una secuencia de sub-elementos -->
    <xs:complexType name="tReparto">
        <xs:sequence>
            <xs:choice minOccurs="1" maxOccurs="unbounded">
                <xs:element name="actor" type="xs:string" />
                <xs:element name="actriz" type="xs:string" />
            </xs:choice>
        </xs:sequence>
    </xs:complexType>

    <!-- Se restringe las URLs a un patrón específico en base a una cadena de carácteres -->
    <xs:simpleType name="url">
        <xs:restriction base="xs:string">
            <xs:pattern value="([Hh][Tt]{2}[Pp][Ss]?)://.*" />
        </xs:restriction>
    </xs:simpleType>
   
</xs:schema>

¿Para qué sirve XSD?

XSD es un fichero de texto plano con extensión .xsd que se utiliza para validar documentos XML, verificando que cumplen con unos requisitos concretos y para asegurar que el documento es válido y lo seguirá siendo en el futuro.

XSD permite validar:
- La jerarquía de los elementos y los atributos permitidos.
- Las restricciones y las reglas de validación para el contenido de los elementos y atributos.
- La descripción de estructuras y tipos de datos complejos.
- La definición de relaciones entre elementos.

Ventajas sobre DTD:
- La sintaxis es más avanzada y proporciona un mayor nivel de detalle.
- Permite restringir con mucha precisión los valores que puede tener un elemento o atributo.
- Los mismos procesadores que permiten interpretar ficheros XML permitirían procesar ficheros XSD ya que este está escrito sobre XML.
- Permite la definición de namespaces.

Pasos de validación:
1. Creamos el documento XSD, también llamado esquema.
2. Vinculamos el documento XSD con el documento XML.

Es importante que las URLs como http://www.w3.org/2001/XMLSchema o http://www.w3.org/2001/XMLSchema-instance deban usar el protocolo http ya que algunas aplicaciones no validan correctamente los documentos XML si indicamos las URLs en su versión https.

Tipos de elementos XML

Hay que tener en cuenta que tipos de elementos existen en XML.

Elementos vacíos:

Son aquellos elementos que no contienen ni sub-elementos, contenido o atributos: 

<etiqueta />

Elementos simples:

Son aquellos elementos que no contienen sub-elementos, aunque pueden tener atributos si se definen mediante simpleContent. Este puede contener solo texto o estar vacío. A continuación se muestra un ejemplo de un elemento simple: 

<etiqueta>Valor de la etiqueta</etiqueta>

Elementos complejos:

Son aquellos elementos que contienen algún atributo y / o otro elemento dentro de él. Cualquier combinación de uno o varios elementos y sub-elementos es válido si el esquema lo permite. A continuación se muestra un ejemplo de un elemento complejo: 

<etiqueta atributo="valor">
    <subetiqueta id="1" num="0011">Valor de la sub-etiqueta</subetiqueta>
</etiqueta>

Se considera como elemento complejo cualquier elemento vacío pero que tenga al menos un atributo, lo mismo si solo tiene un sub-elemento dentro.

Estructura XSD

Los elementos XML que se utilizan para generar un esquema han de pertenecer al namespace XML Schema, el que habitualmente se le asigna el prefijo xs: aunque podría usarse cualquier otro.

<?xml version="1.0" encoding="UTF-8"?>
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <!-- El resto de elementos -->
</xs:schema>

La declaración XML es opcional pero recomendable añadirla al inicio del documento.
La etiqueta schema siempre será el elemento raíz.
El namespace xmlns:xs siempre apuntará a la misma URL http://www.w3.org/2001/XMLSchema.

Vincular un documento XSD a uno XML

En el elemento raíz del documento XML se añadirán los siguientes atributos:

<?xml version="1.0" encoding="UTF-8"?>
<elemento_raiz 
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:noNamespaceSchemaLocation="ruta_documento.xsd"
>
    ...
</elemento_raiz>

xmlns:xsi es la definición de un namespace que se le asigna el sufijo xsi, que puede cambiarse a otro pero debe usarse siempre el mismo en todo el documento. Tiene que tener como valor la URL http://www.w3.org/2001/XMLSchema-instance.

xsi:noNamespaceSchemaLocation es un atributo del namespace xsi utilizado para indicar la ruta al esquema XSD.

Tipos de valores en XSD

Los valores se dividen en 2 grupos principales:

Valores primitivos: No se definen a partir de otros tipos de datos.
Valores derivados: Se definen a partir de otros tipos de datos.

Cada valor puede ser de 2 formas:

Valor predefinido: Valores ya creados por el estándar XML.
Valor personalizado: Valores que podemos crear a partir de uno predefinido.

Imágen con todos los valores existentes en: apache.org

Elemento simple > Validación

A continuación se muestra la estructura para validar un elemento que no contenga ningún atributo ni sub-elementos:

<xs:element name="etiqueta" type="xs:tipo_valor" default="valor" fixed="valor" minOccurs="0" maxOccurs="unbounded"/>

name: Nombre del elemento a validar.
type: Tipo de valor que se permite.
default: Valor por defecto que tendrá si no se especifica uno. No puede usarse a la vez que fixed.
fixed: Valor fijo que se le aplicará al contenido del elemento. No puede usarse a la vez que default.
minOccurs: Cardinalidad que indica el número mínimo de veces que debe aparecer.
maxOccurs: Cardinalidad que indica el número máximo de veces que debe aparecer (unbounded es sin límite).

Ejemplo:

<idioma>Inglés</idioma>
<xs:element name="idioma" type="xs:string" default="Español"/>

Elemento simple > Validación de un dato

Con la siguiente etiqueta se permite definir un tipo de dato simple personalizado a partir de un tipo de dato primitivo o uno personalizado ya definido:

<xs:element name="nombre_elemento">
    <xs:simpleType>
        <!-- Se permiten sub-elementos de tipo: restriction, list o union -->
    </xs:simpleType>
</xs:element>

Si varios elementos o atributos van a tener las mismas restricciones se puede evitar duplicarlo definiéndolo solo una vez y asignándole un nombre personalizable. Este nombre se utilizará para definirlo como type="" en la etiqueta o atributo: 

<xs:element name="etiqueta1" type="NombreRestriccionPersonalizada"/>
<xs:element name="etiqueta2" type="NombreRestriccionPersonalizada"/>
<xs:attribute name="atributo1" type="NombreRestriccionPersonalizada"/>

<!-- Puede declararse el valor personalizado antes o después de los elementos/atributos -->
<xs:simpleType name="NombreRestriccionPersonalizada">
    <!-- Se permiten sub-elementos de tipo: restriction, list o union -->
</xs:simpleType>

Si se hubiera declarado a cada elemento tendríamos mucho más código siendo ineficiente y teniendo duplicidades:

<xs:element name="etiqueta1">
    <xs:simpleType>
        <!-- Se permiten sub-elementos de tipo: restriction, list o union -->
    </xs:simpleType>
</xs:element>
<xs:element name="etiqueta2">
    <xs:simpleType>
        <!-- Se permiten sub-elementos de tipo: restriction, list o union -->
    </xs:simpleType>
</xs:element>
<xs:attribute name="atributo1">
    <xs:simpleType name="NombreRestriccionPersonalizada">
        <!-- Se permiten sub-elementos de tipo: restriction, list o union -->
    </xs:simpleType>
</xs:attribute>

Elemento simple > Lista de elementos del mismo tipo

Con las siguientes 2 maneras se puede definir una lista de elementos del mismo tipo - <xs:list itemType="" />:

<xs:element name="nombre_elemento">
    <xs:simpleType>
        <xs:list itemType="xs:tipo_valor" />
    </xs:simpleType>
</xs:element>

<xs:element name="nombre_elemento" type="lista-enteros" />
<xs:simpleType name="lista-enteros">
    <xs:list itemType="xs:tipo_valor" />
</xs:simpleType>

Por ejemplo, si se indica itemType="xs:integer", el siguiente contenido de la etiqueta sería válido:

<nombre_elemento>1 2 3 4 5 6</nombre_elemento>

Elemento simple > Restricciones

Las restricciones se denominan facetas y se refieren a restricciones que podemos aplicar sobre los valores de los datos de un elemento o atributo. Se debe meter la etiqueta <xs:restriction> dentro de un <simpleType> o dentro de <xs:attribute>. Estructura: 

<xs:simpleType/attribute> <!-- Este es el elemento simple o atributo a restringir -->
    <xs:restriction base="xs:tipo_valor">
        <!-- Restricciones -->
    </xs:restriction>
</xs:simpleType/attribute>

base: El tipo de valor al que aplicaremos esas restricciones.

ℹ️ Las restricciones se explican en apartados posteriores.

Elemento simple > Unión de Tipos

El elemento xs:union permite definir un tipo simple que acepte valores de diferentes tipos de datos.

Ejemplo: Un campo "talla" que acepta un número entero (38, 40) O una talla en letra (S, M, L).

<xs:simpleType name="tipo-talla">
    <xs:union memberTypes="xs:integer talla-letra" />
</xs:simpleType>

<!-- Definimos el tipo personalizado para las letras -->
<xs:simpleType name="talla-letra">
    <xs:restriction base="xs:string">
        <xs:enumeration value="S"/>
        <xs:enumeration value="M"/>
        <xs:enumeration value="L"/>
    </xs:restriction>
</xs:simpleType>

Elemento complejo > Validación

A continuación se muestran 2 formas con la misma estructura para validar un elemento complejo, que puede contener atributos y/o sub-elementos:

<xs:element name="nombre_elemento">
    <xs:complexType>
        <!-- Aquí se definirá su contenido y atributos -->
    </xs:complexType>
</xs:element>
<xs:element name="nombre_elemento" type="elemento_complejo_1" />

<xs:complexType name="elemento_complejo_1">
    <!-- Aquí se definirá su contenido y atributos -->
</xs:complexType>

No se puede usar default en element si se usa complexType.
Con complexType definimos que es un elemento complejo.

Elemento complejo > Secuencia de sub-elementos en un orden específico

Para definir una secuencia de sub-elemento en el orden especificado se utiliza lo siguiente:

<xs:element name="nombre_elemento">
    <xs:complexType>
        <xs:sequence>
            <!-- Aquí se definirá cada sub-elemento -->
        </xs:sequence>
    </xs:complexType>
</xs:element>

Elemento complejo > Secuencia de sub-elementos en cualquier orden

Podemos definir una secuencia de sub-elementos en la que deban aparecer en cualquier orden:

<xs:element name="nombre_elemento">
    <xs:complexType>
        <xs:all>
            <!-- Aquí se definirá cada sub-elemento -->
            <!-- Cada sub-elemento puede aparecer como máximo una vez (maxOccurs="1"). minOccurs solo puede ser 0 o 1. -->
        </xs:all>
    </xs:complexType>
</xs:element>

Elemento complejo > Elección de sub-elementos

Para definir una lista que valide que solo 1 de los sub-elementos aparezcan y no todo se hace con:

<xs:element name="idioma">
    <xs:complexType>
        <xs:choice>
            <!-- Aquí se definirá cada sub-elemento -->
        </xs:choice>
    </xs:complexType>
</xs:element>

Elemento complejo > Contenido simple (Texto + Atributos + Restricciones)

Se utiliza cuando un elemento es complejo porque tiene atributos, pero su contenido es solo texto (sin sub-elementos). Para validar el valor del texto (ej: que sea un número positivo) y añadir atributos, debemos extender un tipo simple personalizado.

Ejemplo: Un precio que debe ser mayor o igual a 0 y tener un atributo de divisa.

<!-- XML OBJETIVO: <precio divisa="EUR">59.65</precio> -->

<xs:element name="precio">
    <xs:complexType>
        <xs:simpleContent>
            <!-- 2. Extendemos la base personalizada para añadirle atributos -->
            <xs:extension base="tipo-precio">
                <xs:attribute name="divisa" type="xs:string" use="required"/>
            </xs:extension>
        </xs:simpleContent>
    </xs:complexType>
</xs:element>

<!-- 1. Primero definimos la restricción del valor (texto/número) -->
<xs:simpleType name="tipo-precio">
    <xs:restriction base="xs:decimal">
        <xs:minInclusive value="0" />
    </xs:restriction>
</xs:simpleType>

simpleContent: Indica que el elemento solo contiene texto/números, no etiquetas hijas.
extension: Permite tomar un tipo existente (en este caso tipo-precio) y "extenderlo" agregándole atributos.
Nota: No se pueden poner restricciones (xs:minInclusive) directamente dentro de una xs:extension. Por eso se crea el simpleType separado primero.

Elemento complejo > Contenido complejo (Explícito vs Abreviado)

Se utiliza complexContent cuando el elemento contiene otros elementos (hijos). Formalmente, todo tipo complejo es una restricción de xs:anyType.

Forma Formal (Explícita):

Así es como XSD entiende internamente la estructura. Se define explícitamente la restricción:

<xs:element name="colores" type="lista-colores"/>

<xs:complexType name="lista-colores">
    <xs:complexContent>
        <xs:restriction base="xs:anyType">
            <xs:sequence>
                <xs:element name="color" type="xs:string" maxOccurs="unbounded" />
            </xs:sequence>
        </xs:restriction>
    </xs:complexContent>
</xs:complexType>

Forma Abreviada (Común):

XSD permite omitir complexContent y restriction si simplemente estamos definiendo una secuencia de hijos estándar. El resultado es idéntico:

<xs:complexType name="lista-colores">
    <xs:sequence>
        <xs:element name="color" type="xs:string" maxOccurs="unbounded" />
    </xs:sequence>
</xs:complexType>

Elemento complejo > Extensión de tipos complejos

Se utiliza cuando queremos reutilizar un tipo complejo existente y agregar nuevos elementos o atributos:

<xs:element name="fichero" type="tipoInformacion" />

<xs:complexType name="tipoInformacion">
    <xs:sequence>
        <xs:element name="nombre" type="xs:string" />
    </xs:sequence>
    <xs:attribute name="formato" type="xs:string" />
</xs:complexType>

<xs:element name="archivo" type="tipoMasdatos" />

<xs:complexType name="tipoMasdatos">
    <xs:complexContent>
        <xs:extension base="tipoInformacion">
            <xs:sequence>
                <xs:element name="creacion" type="xs:dateTime" />
                <xs:element name="edicion" type="xs:dateTime" />
            </xs:sequence>
        </xs:extension>
    </xs:complexContent>
</xs:complexType>

Aquí se crea un nuevo tipo complejo tipoMasdatos que extiende tipoInformacion.
Esto significa que archivo hereda todo lo definido en tipoInformacion (nombre y formato) y añade dos nuevos elementos: creacion y edicion.

La extensión permite reutilizar y ampliar definiciones existentes de manera estructurada, evitando duplicar la información y manteniendo consistencia en el esquema.

Elemento vacío

Un elemento vacío, aunque no tenga contenido, sí puede tener atributos como en el siguiente ejemplo:

<nombre_elemento atributos="" />

Para validar un elemento vacío simplemente hay varias maneras:

<xs:element name="nombre_elemento" />
<xs:element name="nombre_elemento">
    <xs:complexType />
</xs:element>
<xs:element name="precio">
    <xs:simpleType>
        <xs:restriction base="xs:string">
            <xs:maxLength value="0"/>
        </xs:restriction>
    </xs:simpleType>
</xs:element>

En el caso que el elemento sí tenga un atributo pero esté vacío se hace lo siguiente:

<xs:element name="nombre_elemento">
    <xs:complexType>
        <!-- Aquí se definirán los atributos -->
    </xs:complexType>
</xs:element>

Elemento mixto > Cadena de carácteres con sub-elementos

Un elemento mixto es aquel que combina carácteres con sub-elementos como en el siguiente ejemplo:

<nombre_elemento>texto <elemento2>texto</elemento2></nombre_elemento>

Para validarlo, simplemente hay que añadir un atributo mixed al <complexType> como en el siguiente ejemplo:

<xs:element name="nombre_elemento">
    <xs:complexType mixed="true">
        <xs:sequence>
            <!-- Aquí se definirán los sub-elementos -->
        </xs:sequence>
    </xs:complexType>
</xs:element>

Guía Rápida > Combinaciones de Estructura

Resumen visual de las 4 combinaciones posibles de contenido y atributos.

No tiene contenido de texto ni hijos, pero sí atributos. <precio divisa="EUR" />

<xs:element name="precio">
    <xs:complexType>
        <!-- Al no declarar 'sequence' ni 'simpleContent', se asume vacío -->
        <xs:attribute name="divisa" type="xs:string"/>
    </xs:complexType>
</xs:element>

Estructura clásica de anidación. <precio><cantidad>...</divisa>...</precio>

<xs:element name="precio">
    <xs:complexType>
        <xs:sequence>
            <xs:element name="cantidad" type="xs:decimal" />
            <xs:element name="divisa" type="xs:string" />
        </xs:sequence>
    </xs:complexType>
</xs:element>

Combina secuencia y atributos. <precio available="yes"><cantidad>...</precio>

<xs:element name="precio">
    <xs:complexType>
        <xs:sequence>
            <xs:element name="cantidad" type="xs:decimal" />
            <xs:element name="divisa" type="xs:string" />
        </xs:sequence>
        <!-- IMPORTANTE: Los atributos siempre van DESPUÉS de la secuencia -->
        <xs:attribute name="available" type="xs:string" />
    </xs:complexType>
</xs:element>

Reutilización > Grupos de Elementos

El elemento xs:group permite definir un conjunto de elementos comunes para reutilizarlos en varias partes del esquema sin tener que reescribirlos.

1. Definición del grupo (hijo directo de schema):

<xs:group name="datos-basicos">
    <xs:sequence>
        <xs:element name="nombre" type="xs:string" />
        <xs:element name="fecha-nacimiento" type="xs:date" />
    </xs:sequence>
</xs:group>

2. Uso del grupo (dentro de un complexType):

<xs:complexType name="datos-personales">
    <xs:sequence>
        <!-- Referencia al grupo definido anteriormente -->
        <xs:group ref="datos-basicos"/>
        <xs:element name="telefono" type="xs:string"/>
    </xs:sequence>
</xs:complexType>

Reutilización > Grupos de Atributos

El elemento xs:attributeGroup funciona igual que el anterior, pero para agrupar atributos que siempre van juntos.

1. Definición del grupo:

<xs:attributeGroup name="info-archivo">
    <xs:attribute name="formato" type="xs:string"/>
    <xs:attribute name="codificacion" type="xs:string"/>
</xs:attributeGroup>

2. Uso del grupo:

<xs:complexType name="tipo-documento">
    <xs:attributeGroup ref="info-archivo"/>
</xs:complexType>

Atributo > Validación

A continuación se muestra la estructura para validar un atributo de una etiqueta:

<xs:attribute name="etiqueta" type="xs:tipo_valor" default="valor" fixed="valor" use="required/optional/prohibited"/>

name: Nombre del atributo a validar.
type: Tipo de valor que se permite.
default: Valor por defecto que solo se aplica si el atributo no aparece en el XML.
fixed: Valor fijo que se le aplicará al contenido del elemento.
use: Define la obligatoriedad del atributo: obligatorio (required), opcional (optional) o que no pueda aparecer (prohibited).

⚠️ Una vez especificado use="required" no se puede usar la opción default="".

A continuación se muestra un ejemplo de la validación de un elemento y su atributo:

<idioma pais="España">Español</idioma>
<xs:element name="idioma">
    <xs:complexType>
        <xs:simpleContent>
            <xs:extension base="xs:string">
                <xs:attribute name="pais" type="xs:string" default="España"/>
            </xs:extension>
        </xs:simpleContent>
    </xs:complexType>
</xs:element>

Restricciones de longitud

Restricción xs:length:

Partiendo de cualquier tipo de valor se le aplica la restricción de que su longitud debe ser exactamente x (número entero y positivo):

<xs:length value="longitud"/>

Si el valor supera o no llega al valor especificado dará un error al validar el documento.

Restricción xs:minLength:

Partiendo de cualquier tipo de valor se le aplica la restricción de que su longitud mínima debe ser x:

<xs:minLength value="longitud_minima"/>

Restricción xs:maxLength:

Partiendo de cualquier tipo de valor se le aplica la restricción de que su longitud máxima debe ser x:

<xs:maxLength value="longitud_maxima"/>

Restricciones numéricas

Restricción xs:minInclusive:

Partiendo de un tipo xs:integer se le aplica la restricción de que su valor mínimo es x:

<xs:minInclusive value="numero_mínimo"/>

Valores válidos: 12, +12, 1.2, +1.2

Restricción xs:maxInclusive:

Partiendo de un tipo xs:integer se le aplica la restricción de que su valor máximo es x:

<xs:maxInclusive value="numero_máximo"/>

Se puede usar las restricciones xs:minInclusive y xs:maxInclusive acortando las posibilidades a un rango de números.

Restricción xs:minExclusive:

Partiendo de un tipo xs:integer se le aplica la restricción de que su valor mínimo permitido es mayor que x:

<xs:minExclusive value="numero_antes_del_mínimo"/>

Ejemplo: Si x es 10, el valor mínimo permitido es 11

Restricción xs:maxExclusive:

Partiendo de un tipo xs:integer se le aplica la restricción de que su valor máximo permitido es menos que x:

<xs:maxExclusive value="numero_despues_del_máximo"/>

Ejemplo: Si x es 10, el valor máximo permitido es 9

Restricción xs:totalDigits:

Partiendo de un tipo xs:integer se le aplica la restricción de que el número puede tener hasta x dígitos, siendo x un número positivo:

<xs:totalDigits value="digitos_máximos"/>

Restricción xs:fractionDigits:

Partiendo de un tipo xs:decimal se le aplica la restricción de que el número puede tener hasta x dígitos después del punto decimal:

<xs:fractionDigits value="digitos_decimales_máximos"/>

Restricciones con opciones válidas

Restricción xs:enumeration:

Asignamos una lista de uno o varios elementos xs:enumeration cuyo valor sea cada opción permitida:

<xs:enumeration value="valor_permitido_1"/>
<xs:enumeration value="valor_permitido_2"/>
...

Si un valor no coincide con los permitidos se producirá un error al validar el documento XML.

Se recomienda tener una lista breve para tener un documento lo más eficiente posible.

Restricciones sobre espacios en blanco

Restricción xs:whiteSpace:

Con esta restricción se valida los espacios en blanco, tabulaciones, saltos de línea y retornos de carro que haya en un valor:

<xs:whiteSpace value="preserve/replace/collapse"/>

preserve: No se realizan modificaciones (mantiene todo igual).

replace: Sustituye todas las tabulaciones, saltos de línea y retornos de carro por espacios en blanco.

collapse: Realiza un replace y además elimina espacios extremos y reduce varios espacios consecutivos a un único espacio en blanco.

Restricciones con Expresiones regulares

Una expresión regular es una secuencia de caracteres que se utiliza para definir un patrón. También se conocen como regex (regular expression).

A continuación se muestra la sintaxis de como aplicar un patrón a un elemento o atributo para filtrar so contenido o valor:

<xs:pattern value="patrón"/>

Sintaxis para crear expresiones regulares
PATRÓN DESCRIPCIÓN VALORES VÁLIDOS 
ola
 Buscar una cadena de caracteres en un valor. ola, hola, volar, etc. 
^X
 El valor debe empezar con la cadena indicada. XML, XSD, XOGAR, etc. 
$r
 El valor debe acabar con la cadena indicada. Dormir, Buscar, Encontrar, etc. 
x*
 Un carácter coincide de 0 a n veces. Todos los valores que contengan o no la "x". 
X+
 Un carácter coincide de 1 a n veces. XML, XXZ, etc. 
X?
 Un carácter coincide 0 o 1 vez. XML, XSD, etc. 
.
 Permite cualquier carácter (excepto salto de línea). A, B, etc.
O usar .* para cualquier carácter sin límite.		 
\.
 Buscar un carácter literal, ignorando su significado especial. . 
[aeiou]
 Lista de caracteres que coincidan (solo uno de ellos). a, e, i, o, u 
[^aei]
 Lista de caracteres que no coincidan. Todos los valores excepto a, e, i 
[a-z]
 Rango de caracteres que coincidan. Todos los caracteres de la a a la z 
c{1,3}
 Un carácter puede repetirse entre min y max veces. c, cc, ccc 
\bhola\b
 Coincidir palabra exacta. hola 
\d
 Filtrar números del 0 al 9. 0-9 (ej: 12, 65, etc.) 
\w
 Filtrar cualquier carácter de palabra (letra, número, guion bajo). A, b, 1, _, etc. 
\W
 Filtrar cualquier carácter que no sea de palabra. !, @, espacio, etc. 
patrón1 | patrón2
 Elegir entre 2 patrones. Valores que coincidan con patrón1 o patrón2 
patrón1 (?=patrón2)
 Busca patrón1 solo si le sigue patrón2. Valores que cumplan patrón1 seguido de patrón2

Expresiones regulares > Herramientas

Las web regexr.com o regex101.com sirven para practicar la creación de estos patrones.
