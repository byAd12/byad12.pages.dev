Básico

Sintaxis de Comandos

prompt>comando argumentos

prompt>: Indica el hostname del equipo que estamos configurando, aparte de si tenemos permisos: #, o no: >.
comando: Comando a ejecutar.
argumentos: Antes de especificar una palabra clave o argumentos, se añade un espacio después del comando.

Reglas de Nomenclatura (Hostnames)

Reglas: Comenzar con una letra. No contener espacios. Finalizar con letra o dígito. Utilizar únicamente letras, dígitos y guiones. Tener menos de 64 caracteres de longitud.

Ejemplo válido: 

Sw-Floor-1

Para poder cambiar el hostname de un equipo Cisco se usa el siguiente comando:

# hostname <Nombre>

Reglas de Contraseñas

Reglas: Tener más de 8 caracteres de longitud. Combinaciones de letras, números y símbolos. Evitar el uso de la misma en varios dispositivos. No usar palabras comunes.

Generador de contraseñas: letpass.com

Gestión de la configuración

Configuración Inicial y Ayuda

Si escribimos un comando no válido intentará resolver ese texto como DNS. Este proceso lleva tiempo y se puede saltar mediante las teclas:

CTRL + BLOQ MAYUS + 6

Habilitar permisos de EXEC privilegiado en el equipo. El prompt se modificará de > a #:

> enable

Muestra todos los comandos disponibles según donde te encuentres:

# ?

Se puede poner un comando seguido de ? para enseñar sus palabras claves y atributos disponibles. Además puedes buscar una palabra completa poniendo la(s) letra(s) por la(s) que empieza seguido de ?.

Para configurar la fecha y hora del equipo:

# clock set hora:min:seg día mes año

Mes: Debe ser el nombre con letras y en inglés.

Luego podemos ver que hora tiene configurado el equipo con:

# show clock

Gestión de Archivos de Configuración

Para ver las configuraciones del dispositivo debemos estar en la raíz del dispositivo y en modo EXEC privilegiado.

El siguiente comando nos permite ver configuraciones (interfaces, login y contraseñas) que están guardadas temporalmente y se eliminan al reiniciar el equipo:

# show running-config

Para ver la configuración que persiste en NVRAM incluso al reiniciar el equipo:

# show startup-config

Para guardar la memoria temporal en NVRAM se utiliza wr o:

# copy running-config startup-config

Para eliminar la memoria temporal se utiliza:

# reload

Por último, para eliminar la memoria NVRAM se utiliza:

# erase startup-config

Sistema Operativo IOS

Para cambiar de la variable de entorno BOOT la ruta al archivo de imagen se utiliza el siguiente comando:

(config)# boot system flash:/<SistemaArchivos>/<NombreArchivoIOS>.bin

Eliminar toda la configuración del sistema

Eliminaremos la configuración de arranque y el fichero de la configuración de VLANs:

# erase startup-config
# delete vlan.dat
# write erase

Cuando pregunte si estamos seguros debemos responder con y seguido de ENTER.

Herramientas

Herramientas de conectividad

Se puede enviar un paquete ICMP a un host de destino para verificar si responde:

ping <IP_Destino>

El siguiente comando permite identificar los routers que hay entre el origen y destino:

traceroute <IP_Destino>

Para identificar estos routers, el equipo envía el mismo paquete subiendo el TTL (Time To Live) por cada uno.

Este otro comando permite ver los dispositivos Cisco que están conectados directamente:

# show cdp neighbors

Herramientas de acceso remoto

Para conectarse mediante Secure Shell a un host remoto:

ssh -l <Usuario> <IP_Destino>

Para conectarse mediante Telnet a un host remoto:

telnet <IP_Destino>

Nota - TELNET y SSH:

Si en el router o switch que queremos conectarnos no tiene en las líneas vty 0 15 un password y login, el destino rechazará la conexión automáticamente.

⚠️ Nota - Problema de seguridad en TELNET:

El tráfico de Telnet se envía en texto plano, incluyendo usuario y contraseña, lo que lo hace vulnerable a ataques man-in-the-middle. Demostración en Packet Tracer:

1. Entra en el Command Prompt de un PC e inicia una conexión:

C:\> telnet (-l <Usuario>) <IP_Destino>

2. Antes de ingresar la contraseña, cambia Packet Tracer de Realtime a Simulation.

3. Se puede apreciar que por cada letra que tenga la contraseña se crea un paquete de parte del cliente. Abre a uno desde la sección Event List.

4. En la nueva ventana pasa al modo Outbound PDU Details y baja hasta el apartado TELNET. Se puede ver que en cada paquete se enseña la letra enviada:

TELNET DATA:C

Filtrar la salida de un comando

Sintaxis para filtrar:

<Comando> | <Filtro> <Palabra(s) a filtrar>

FILTROS:

Para filtrar una sección entera mediante el nombre de esta sección como vty, ospf, ...:

show running-config | section line vty

Para filtrar solo las líneas que contengan una palabra como up:

show ip interface brief | include up

Para devolver solo las líneas que no contengan una palabra como unassigned:

show ip interface brief | exclude unassigned

Para devolver todo lo que haya a continuación de la primera línea que contenga una palabra como "Gateway":

show ip route | begin Gateway

Comandos de Verificación

Comandos de verificación > Importante

El comando show solo funciona si se ejecuta fuera de (config)# como el siguiente ejemplo:

# show ip interface brief

Si se desea ver el comando dentro de la configuración hay que añadir la palabra do al principio del comando como el siguiente ejemplo:

(config)# do show ip interface brief

Tenga en cuenta que al usar do no se podrá usar el autocompletado de opciones del comando.

Comandos de verificación > Interfaces

Para ver todas las configuraciones de las interfaces del dispositivo en IPv4:

show ip interface brief
show ipv6 interface brief

Enseña de cada interfaz el nombre, IP, Si está reconocida por el sistema (OK?), estado y protocolo. Ejemplo:

Router# show ip interface brief
Interface              IP-Address      OK? Method Status                Protocol
FastEthernet0/0        192.168.1.1     YES manual up                    up
FastEthernet0/1        unassigned      YES unset  administratively down down
Vlan1                  unassigned      YES unset  administratively down down

Para ver todas las estadísticas y estado de todas las interfaces del equipo:

show interfaces

Este comando es útil para detectar problemas comunes de medios. Una de las partes más importantes de esta salida es la visualización de la línea y el estado del protocolo de enlace de datos.

También se puede ver las estadísticas y estado de una interfaz en específico:

show interfaces <Interfaz>

Esto puede darnos información sobre errores de tramas gigantes, colisiones, runts, desbordamiento, etc.

Los siguientes comandos muestran información de IPv4 o IPv6 de todas las interfaces:

show ip interface brief
show ipv6 interface brief

Los siguientes comandos muestran información de IPv4 o IPv6 de una interfaz en específico:

show ip interface <Interfaz>
show ipv6 interface <Interfaz>

En ipv6 se muestra todas las direcciones IPv6 que pertenecen a la interfaz (Joined group address(es)).

Se puede ver los comandos aplicados a una interfaz específica que se están aplicando actualmente:

show running-config interface <Interfaz>

Para ver la tabla ARP IPv4:

show ip arp

Para ver la tabla de vecinos IPv6:

show ipv6 neighbors

Para ver la tabla MAC se puede usar las 2 siguientes formas:

show mac address-table
show mac-address-table

Comandos de verificación > Sistema

Se puede ver el estado del hardware e información del sistema con:

show version

Para ver el contenido de la memoria flash flash::

Configuración básica

Navegación y configuración de interfaces

Podemos entrar en modo de configuración global del dispositivo de 2 formas:

# configure terminal
# config t

Podemos configurar un puerto de switch o una interfaz de red de router (solo si entramos en config anteriormente):

(config)# interface <Interfaz> <Número_Interfaz>

Interfaz y Número: Podemos especificar un nombre completo como FastEthernet0/1 o su abreviatura como fa0/3. 0/3 Significa que de la interfaz 0, se elige el puerto 3.

Podemos asignar una descripción a cada interfaz:

(config-if)# description <Descripción>

Para configurar parámetros de consola co, SSH y Telnet vty o acceso auxiliar:

(config)# line <co/vty> <Número/Rango>

Número: El número empieza desde 0 e identifica el puerto a configurar. Rango: Si se quiere configurar varias líneas a la vez, como las 16 líneas de vty se usa un rango como: 0 15, con un espacio entre ambos números.

Habilitar una interfaz después de realizar un cambio:

(config-if)# no shutdown

O deshabilitarla:

(config-if)# shutdown

Podemos salir de cualquier modo de configuración al anterior. Ej. Si nos encontramos en (config-if) y enviamos este comando pasamos a (config):

(config-if)# exit
(config)# exit
# exit
> <Comandos>

O podemos salir de cualquier modo de configuración al modo inicial:

(config-if)# end
# <Comandos>

Direccionamiento IPv4 e IPv6

Si es un switch hay que configurar IPs mediante VLANs se usa (la VLAN por defecto es 1):

(config)# interface vlan <VLAN_ID>

Podemos asignar a una interfaz una IPv4 mediante:

(config)# interface <Interfaz> <Número_Interfaz>
(config-if)# ip address <IPv4> <Máscara>
(config-if)# no shutdown

Nota: Hay que recordar poner la configuración del default-gateway y name-server.

O además una IPv6:

(config)# interface <Interfaz> <Número_Interfaz>
(config-if)# ipv6 address <IPv6/Máscara>
(config-if)# no shutdown

Si se trata de una dirección IPv6 LLA fe80::/10 se modifica el comando a:

(config-if)# ipv6 address <LLA_IPv6> link-local

Default Gateway (Switch)

Para poner la IP del gateway por defecto hay que ponerlo en la configuración global:

(config)# ip default-gateway <IpGateway>

Servicios DNS

Asignar un nombre de dominio (ej. pargapondal.inf):

(config)# ip domain-name <Dominio>

Desactivar la búsqueda de DNS:

(config)# no ip domain-lookup

Para asignar un servidor DNS a un switch o router hay que ponerlo en la configuración global:

(config)# ip name-server <IP_Servidor_DNS>

Seguridad

Filtro de protocolos (Transport Input)

Podemos permitir ciertos protocolos de acceso remoto con:

(config)# line vty 0 15
(config-line)# transport input <ssh telnet>

Podemos permitir tanto ssh y telnet o solo uno de los dos. Se separan mediante espacios.

Seguridad básica y banners

Poner una contraseña en líneas de acceso:

(config)# line <console/vty> <NumPuerto/Rango>
(config-line)# password <Contraseña>

Para poner una contraseña al comando enable (texto plano):

(config)# enable password <Contraseña>

Para ocultar la contraseña:

(config)# service password-encryption

Aunque oculta el contenido, se puede descifrar fácilmente debido a su débil cifrado con herramientas como firewall.cx.

Para poner una contraseña al comando enable (encriptando con MD5):

(config)# enable secret <Contraseña>

Si se tenía enable password anteriormente se ignorará y solo se usará enable secret.

Aplicar una política de usuario y contraseña: authentication, o solo contraseña: local:

(config-line)# login <authentication/local>

Poner un banner que se enseñará en la página de login:

(config)# banner motd <Carácter>

Carácter: Indicamos un carácter que no se vaya a repetir en el texto. Después de ejecutarlo pondrá un diálogo para poner un texto personalizado que verán al iniciar sesión. Acabaremos el comando con el carácter especificado y luego ENTER.

Bloquear un rango de puertos no usados:

(config)# interface range <Interfaz/Rango>, <Interfaz/Rango>, ...

Interfaz/Rango: Podemos seleccionar varios rangos de interfaces y aplicar los mismos cambios a todos. Ejemplo: F0/2-24, G0/2 (F0/2 a F0/24 y G0/2).

(config-if-range)# shutdown

Gestión de usuarios y privilegios

Para crear un usuario sin privilegios:

(config)# username <Nombre_Usuario> secret <Contraseña>

Para crear un usuario con privilegios:

(config)# username <Nombre_Usuario> privilege <TipoPrivilegio> secret <Contraseña>

TipoPrivilegio: Número del 0 al 15 que indica el nivel de privilegios que tendrá el usuario:

- Del 2 al 14 son usuarios con permisos parciales.
- El 15 equivale al comando enable.
- El 1 equivale a un usuario normal.
- El 0 equivale a un acceso muy restringido.

Para aplicar una directiva de autentificación por usuario y contraseña:

(config)# line vty 0 15
(config-if)# login local

Políticas de seguridad (bloqueos)

Para aplicar una longitud mínima a cada contraseña que se cree desde este momento (no se aplica a contraseñas ya creadas):

(config)# security passwords min-length <NúmeroMínimo>

Para cerrar la sesión de un usuario después de x minutos:

(config)# line vty 0 15
(config-if)# exec-timeout <Minutos>

Bloquear a cualquier persona durante x minutos que no pueda iniciar sesión después de x intentos dentro de un período de x minutos:

(config)# login block-for <SegundosBloqueo> attempts <Intentos> within <PeriodoMinutos>

Política de avisos del sistema

Cisco IOS suele lanzar alertas mientras el usuario escribe, cortando el comando en algunos casos. Se puede evitar con el siguiente comando, que solo lanza avisos cuando el usuario no haya escrito nada:

(config)# interface console/vty <Número_Interfaz>
(config-line)# logging synchronous

Se suele configurar en las interfaces de consola o vty, para acceso remoto.

Configuración de un servidor SSH

1. Antes de nada hay que verificar si el switch soporta realizar funciones criptográficas. Para ello debe de ejecutar este comando sin errores: 

# show ip ssh

2. Poner un hostname único y diferente al de por defecto.

3. Asignar un nombre de dominio (ej. pargapondal.inf):

(config)# ip domain-name <Dominio>

4. Generar una clave de 4096 Bytes u otro tamaño con el algoritmo RSA:

(config)# crypto key generate rsa general-keys modulus 4096

Permitir solo el tráfico SSH en las líneas de acceso remoto:

(config)# line vty 0 15
(config-if)# transport input ssh

Desde el momento de poner el comando se denegará conexiones telnet, solo admitirá SSH. Si se tenía alguna conexión telnet existente no la cierra.

Hay que recordar de poner login local si se va a permitir el acceso con nombre de usuario y contraseña.

🛡️ Para reforzar la seguridad, se puede forzar al switch a realizar únicamente conexiones por SSH v2.0:

(config)# ip ssh version 2

Configuración específica

Configuración de full-duplex en una interfaz

Primero entraremos a la interfaz deseada y asignaremos el modo full-duplex:

(config)# interface <Interfaz>
(config-vlan)# duplex full

Configuración de velocidad en una interfaz

Primero entraremos a la interfaz deseada y asignaremos la velocidad en Mbps:

(config)# interface <Interfaz>
(config-vlan)# speed <VelocidadEnMbps>

Configuración de Auto-MDIX en una interfaz

Primero entraremos a la interfaz deseada y asignaremos que funcione como Auto-MDIX:

(config)# interface <Interfaz>
(config-vlan)# mdix auto

Podemos comprobar que se haya aplicado Auto-MDIX en la interfaz especificada con el comando:

# show controllers ethernet-controller <Interfaz> phy | include MDIX

Si es correcto debería de devolver la siguiente línea:

Auto-MDIX    :  On    [AdminState=1    Flags=0x00052248]

Creación de una interfaz loopback

Una interfaz de loopback en un router no permite conectar otros dispositivos y sirve para que una interfaz siempre esté encendida (siempre que el router esté en funcionamiento). Por ejemplo, se puede usar con fines de prueba, como la prueba de procesos de routing interno, mediante la emulación de redes detrás del router.

Para crear una, simplemente se ejecuta el siguiente comando seguido de la asignación de una IP:

(config)# interface loopback <Número>
(config-if)# ip address <IP> <Máscara>

(config)# vlan <VLAN_ID>

(config-vlan)# name <NombreVLAN>

(config)# delete flash:vlan.dat

Otra forma abreviada es quitando el dispositivo flash de la ruta:

(config)# delete vlan.dat

VLAN > Visualización de puertos asignados

Con el siguiente comando se puede ver qué VLANs se asignaron a ciertos puertos:

# show vlan brief

Por defecto se puede ver que todas las interfaces del switch se asignaron a la VLAN 1 (predeterminada).

Con el siguiente comando se muestra la información de una VLAN específica:

# show vlan id <VLAN_ID>

# show interfaces <Interfaz> switchport

VLAN > Asignación de una IP

Simplemente entraremos a la interfaz con interface y le asignaremos la configuración necesaria:

(config)# interface vlan <VLAN_ID>

Switchport

Modos de interfaz negociados (Switchport)

El comando switchport mode tiene opciones adicionales para negociar el modo de interfaz. Sintaxis del comando:

Switch(config)# switchport mode {  access | dynamic {  auto | desirable } |  trunk }
Modos de switchport 
access
 Pone la interfaz (puerto de acceso) en modo permanente de no trunking y negocia para convertir el enlace en un enlace no troncal. La interfaz se convierte en una interfaz no troncal, independientemente de si la interfaz vecina es una interfaz troncal. 
dynamic auto
 La interfaz se convierte en una interfaz de enlace troncal, si la interfaz vecina está configurado en modo troncal o deseable. 
dynamic desirable
 La interfaz se convierte en una interfaz de enlace troncal, si la interfaz vecina está configurado en modo automático troncal, deseable o dinámico. 
trunk
 El puerto queda configurado en modo troncal de forma permanente y negocia para que el enlace se convierta en una conexión troncal. La interfaz se convierte en una interfaz de enlace troncal, incluso si la interfaz vecina no es una interfaz troncal.

Verificación del modo de DTP

# show dtp interface <Interfaz>

Negociación

Habilitar la negociación de modos entre puertos

(config)# interface <Interfaz>
(config-if)# switchport mode trunk
(config-if)# switchport nonegotiate

Desactivar la negociación de modos entre puertos

(config)# interface <Interfaz>
(config-if)# switchport mode dynamic auto

Spanning-tree

Spanning-tree > Asignar un switch como raíz

Dentro del switch que queramos definir como raíz ejecutaremos lo siguiente:

(config)# spanning-tree vlan <VLAN_ID> root primary

VLAN_ID: La VLAN en la que este switch será raíz.

Para verificarlo ejecutaremos el siguiente comando:

(config)# do show spanning-tree vlan <VLAN_ID>

En el apartado Root ID > Priority se puede observar que el valor es 4096 menor que la prioridad del switch que anteriormente actuaba como raíz, lo que confirma que el nuevo switch ha sido elegido como raíz.

Nota Teórica: Este comando no fija una prioridad estática simple. El switch verifica la prioridad del raíz actual y configura su propia prioridad para ser 4096 puntos menor (si es posible) para ganar la elección. Si el raíz actual tiene 24576, este switch bajará a 20480.

Spanning-tree > Asignar un switch como secundario

Dentro del switch que queramos definir como secundario ejecutaremos lo siguiente:

(config)# spanning-tree vlan <VLAN_ID> root secondary

VLAN_ID: La VLAN en la que este switch será secundario.

Para verificarlo ejecutaremos el siguiente comando:

(config)# do show spanning-tree vlan <VLAN_ID>

En el apartado Bridge ID > Priority se puede observar que el valor no es menor que el de Root ID > Priority pero si es menor que el de otros switches en esa VLAN.

Nota Teórica: Este comando fija la prioridad exactamente en 28672. Si todos los demás switches tienen la prioridad por defecto (32768), este switch "secundario" se convertirá en el Raíz (porque 28672 < 32768). Solo actuará como secundario si existe otro switch con prioridad "Primary" (ej. 24576).

Spanning-tree > Conceptos de prioridad

La prioridad del bridge se compone de:
Prioridad Total = Prioridad Base + ID de VLAN

Los valores cambian en bloques de 4096. Los valores típicos son:

Por defecto 32768 + VLAN
Root Secondary 28672 + VLAN
Root Primary 24576 + VLAN (o 4096 menos que el raíz actual)

Nota de Desempate: Si dos switches tienen la misma prioridad (ej. ambos por defecto), gana el que tenga la dirección MAC más baja.

EtherChannel - LACP

EtherChannel > LACP > Configuración

Antes de realizar los siguientes pasos, asegúrese de que las VLANs ya estén creadas y que las demás interfaces necesarias estén correctamente configuradas en ambos switches.

Paso 1. Configurar el EtherChannel:

(config)# interface Po<Número>
(config-if)# switchport mode trunk
(config-if)# ...

En esta interfaz lógica se debe aplicar exactamente la misma configuración que se usaría en las interfaces físicas que formarán el EtherChannel.

El número es un identificador que representa al EtherChannel y debe coincidir con el que se usará en los pasos posteriores.

Paso 2. Aplicar la misma configuración del paso anterior en las interfaces físicas que se agruparán en el EtherChannel:

(config)# interface range <Interfaz1,Interfaz2,...>
(config-if-range)# switchport mode trunk
(config-if-range)# ...

Todas las interfaces deben tener configuraciones idénticas para que el EtherChannel funcione correctamente.

Paso 3. Apagar las interfaces:

(config)# interface range <Interfaz1,Interfaz2,...>
(config-if-range)# shutdown

Este paso evita inconsistencias o errores durante la creación del EtherChannel.

Paso 4. Agregar las interfaces al EtherChannel:

(config)# interface range <Interfaz1,Interfaz2,...>
(config-if-range)# channel-group <Número> mode active

El número debe coincidir con el identificador del EtherChannel configurado en el paso 1.

El modo active indica que se utilizará LACP para la negociación del enlace.

Importante: Antes de continuar, asegúrese de haber ejecutado correctamente todos los pasos anteriores en ambos switches.

Paso 5. Encender las interfaces:

(config)# interface range <Interfaz1,Interfaz2,...>
(config-if-range)# no shutdown

Una vez habilitadas, las interfaces deberían formar el EtherChannel correctamente.

Nota: Una vez creado el EtherChannel, cualquier cambio que se vaya a hacer a las interfaces se debe hacer solo al grupo Po<Número> mediante el comando:

(config)# interface Po<Número>
(config-if)# ...

EtherChannel > LACP > Verificación

Para verificar si el EtherChannel se creó correctamente hay que ejecutar el siguiente comando en ambos switches:

# show etherchannel summary

Importante: En el apartado "Ports" debe aparecer una P como en el siguiente ejemplo:

# show etherchannel summary
    Number of channel-groups in use: 1
    Number of aggregators:           1

    Group  Port-channel  Protocol    Ports
    ------+-------------+-----------+------------------
    1      Po1(SU)           LACP   Gig0/1(P) Gig0/2(P)

Otra manera de verificarlo es mirar si la interfaz port-channel está encendida:

# show interfaces port-channel <Número>

El número es un identificador que representa al EtherChannel.

Paso 1. Entrar a las interfaces y eliminar el channel-group:

(config)# interface range <Interfaz1,Interfaz2,...>
(config-if-range)# no channel-group <Número>

Paso 2. Eliminar el port-channel:

(config)# no interface port-channel <Número>

El número del port-channel es el mismo que se le asignó a la interfaz EtherChannel Po<Número>.

VLAN Trunking Protocol

VTP > Configuración en el switch servidor

1. Meteremos al switch en un dominio VTP:

(config)# vtp domain <Nombre_Dominio>

2. Asignaremos una contraseña necesaria para que otros switches tengan que introducirla para conectarse al dominio:

(config)# vtp password <Contraseña>

3. Asignaremos al switch que actúe en modo servidor:

(config)# vtp mode server

4. Por seguridad, asignaremos que el switch use solo VTP versión 2:

(config)# vtp version 2

⚠️ Advertencia de Seguridad (Dominio NULL):

Por defecto, un switch no tiene nombre de dominio (es NULL). Si conectas un switch nuevo a la red sin configurar el dominio primero, este adoptará automáticamente el nombre del dominio del primer paquete VTP que reciba. Esto puede causar que el switch se una a la red equivocada y altere las VLANs.

VTP > Meter a un cliente en un dominio

1. Meteremos al switch en el mismo dominio VTP que el servidor:

(config)# vtp domain <Nombre_Dominio>

2. Asignaremos la contraseña que se definió en el servidor:

(config)# vtp password <Contraseña>

3. Por seguridad, asignaremos que todo el switch use solo VTP versión 2:

(config)# vtp version 2

4. Definir en qué modo va a trabajar el switch:

(config)# vtp mode <client/transparent>

client: En este modo solo se podrá cambiar la lista de VLANs cuando se reciba comunicaciones de otros equipos.
transparent: En este modo el dispositivo actuará como un dispositivo aislado, sin compartir sus VLANs con el resto de dispositivos.

VTP > Ver la configuración

Para verificar el estado de VTP y la configuración usaremos el siguiente comando:

# show vtp status

Esto mostrará todas las configuraciones anteriores menos la contraseña, además de un Configuration Revision que es muy importante al indicar la cantidad de cambios que se realizaron en este switch (por defecto es 0).

En cualquier otro switch que tenga un Configuration Revision menor al switch servidor, tendrá que actualizar su lista de VLANs.

VTP > Ver la contraseña del dominio

Cualquier switch de un mismo dominio debe tener la misma contraseña VTP que indique el siguiente comando:

# show vtp password

En versiones antiguas de IOS este comando no existía o la contraseña venía oculta en show vtp status.

VTP > Reiniciar la revisión (seguridad)

Importante: Antes de conectar un switch a la red, debes asegurar que su "Configuration Revision" sea 0.

Si conectas un switch con una revisión mayor a la del servidor, este sobrescribirá y borrará las VLANs de toda la red.

Método 1 - Cambiar el modo a Transparente y volver:

Al pasar a modo transparente, el contador de revisión se reinicia a 0.

(config)# vtp mode transparent
(config)# vtp mode client

Método 2 - Borrar la base de datos de VLANs:

# delete vlan.dat
# reload

Hot Standby Router Protocol

HSRP > Configuración de una IP

En cada switch de capa 3 o router en el que se vaya a compartir una IP en caso de que un dispositivo caiga, realizaremos los siguientes pasos:

1. Entraremos la interfaz física o lógica (vlan) a la que configuraremos la IP en HSRP:

(config)# interface <Interfaz>

2. Le asignaremos una IP diferente a la que se va a configurar con HSRP:

(config-if)# ip address <IP_Secundaria>

3. Asignaremos que se use solo la versión 2 de HSRP:

(config-if)# standby version 2

4. Ahora meteremos a la interfaz en un grupo y le asignaremos que IP compartirá con otro(s) dispositivo(s):

(config-if)# standby <Grupo> ip <IP_HSRP>

Grupo: Número que vincula lógicamente a varios equipos para que funcionen como un único equipo virtual.
IP_HSRP: La dirección IPv4 que compartirá con otros dispositivos en el mismo grupo.

5. Por último encenderemos la interfaz si no se ha hecho anteriormente:

(config-if)# no shutdown

HSRP > Configuración de la prioridad

1. Entraremos la interfaz física o lógica (vlan) a la que configuraremos la prioridad:

(config)# interface <Interfaz>

2. Le asignaremos una prioridad numérica con el siguiente comando:

(config)# standby <Grupo> priority <Prioridad>

Grupo: Número que vincula lógicamente a varios equipos para que funcionen como un único equipo virtual.
Prioridad: Número de 0 a 255 que indicará qué prioridad tenga el dispositivo. Cuanto mayor sea, mayor preferencia tendrá para ser el dispositivo activo.

Nota: No hace falta definir que tiene prioridad 100 ya que es la prioridad por defecto que tienen todos los switches.

HSRP > Configuración de preempt

Con preempt podemos forzar el cambio de cesión de IP de un dispositivo a otro si hay un cambio de prioridad.

1. Entraremos la interfaz física o lógica (vlan) a la que configuraremos el modo:

(config)# interface <Interfaz>

2. Le asignaremos que funcione en modo preempt:

(config)# standby <Grupo> preempt

Grupo: Número que vincula lógicamente a varios equipos para que funcionen como un único equipo virtual.

Nota: No hace falta poner preempt en el switch que no será el activo.

En el comando show standby brief debería aparecer la interfaz que el modo preempt está activo (P).

HSRP > Verificación de la configuración

Para verificar el estado de HSRP y la configuración de cada interfaz con IP compartida usaremos el siguiente comando:

# show standby

Este comando muestra detalles que el comando con brief omite, como la MAC virtual activa, los temporizadores por defecto (Hello 3s, Hold 10s) y el contador de cambios de estado del router.

También se puede ver en formato de resumen:

# show standby brief

Dynamic Host Configuration Protocol v4

DHCPv4 > Servidor > Configuración

1. Primero hay que excluir todas las IPs que sean reservadas/estáticas del rango de IPs que DHCP va a conceder:

Para excluir una única IP:

(config)# ip dhcp excluded-address <IP>

Para excluir un rango de IPs:

(config)# ip dhcp excluded-address <IP_Inicio> <IP_Fin>

2. Ahora hay que definir un nombre identificativo para el pool de DHCPv4 que hay se debe crear:

(config)# ip dhcp pool <Nombre_Pool>

3. Una vez ejecutemos el paso 2 entraremos en un modo donde podremos configurar las configuraciones que se darán a los clientes:

Configuración obligatoria Explicación 
network <Direccion_Red> <Máscara>
 Sirve para definir la máscara de red en formato punteado o CIDR. 
default-router <IP>
 Importante, es la IP de la puerta de enlace / default gateway. 
dns-server <IP>
 Es la IP del servidor DNS. Se puede omitir si no se quiere configurar pero, cuando sea necesario, habrá que configurarlo manualmente.
Configuración opcional Explicación 
domain-name <Dominio>
 Es el nombre de dominio que tendrán los clientes DHCP. 
lease {días [horas] [minutos] | infinite}
 Es la duración de una concesión de IP. Ejemplo: "lease 1 12 25". 
netbios-name-server <IP>
 Es el servidor WINS que ejecute el servicio NetBIOS.
Se recomienda reemplazar su uso por DNS.

DHCPv4 > Servidor > Configurar el modo de reenvío

Seguridad - Spanning-tree

Spanning-tree > Activar Port-Fast

Primero entraremos en la interfaz deseada y activaremos mediante spanning-tree el modo portfast:

(config)# interface <Interfaz>
(config-if)# spanning-tree portfast

Esto evitará que la interfaz espere alrededor de 50 segundos para empezar a aceptar tramas.

Importante: Debe ponerse solo en puertos que estén en modo acceso.

Además podemos habilitar portfast por defecto en todos los puertos del switch que están en modo acceso:

(config)# spanning-tree portfast default

Spanning-tree > Activar BPDU-Guard

Primero entraremos en la interfaz deseada y activaremos mediante spanning-tree el modo bpduguard:

(config)# interface <Interfaz>
(config-if)# spanning-tree bpduguard enable

Esto pondrá la interfaz en estado err-disable si recibe una BPDU, para evitar que se conecte un switch y provoque bucles en la red.

Importante: Debe ponerse solo en puertos que estén en modo acceso y en portfast ya que en los puertos en modo troncal suelen circular mensajes STP, ya que en los puertos en modo troncal suelen circular BPDUs, lo que provocaría que la interfaz entrase en err-disable.

Además podemos habilitar BPDU-Guard por defecto en todos los puertos del switch que están en modo acceso:

(config)# spanning-tree portfast bpduguard default

Siempre hay que activar portfast si se usa BPDU-Guard.

Spanning-tree > Verificación

Con el siguiente comando podremos ver toda la configuración relacionada con STP:

# show running-config | begin span

También podemos ver un resumen de la configuración de STP mirando si portfast o BPDU-Guar está habilitado:

# show spanning-tree summary

Seguridad - Problemas de reconocimiento

(Des)habilitar el protocolo CDP

CDP es un protocolo habilitado por defecto en todos los switches de Cisco. Puede detectar automáticamente otros dispositivos con CDP habilitado y ayudar a configurar automáticamente la conexión.

Problema de seguridad: La información de CDP se envía por los puertos con CDP habilitado en transmisiones periódicas sin encriptar, incluyendo datos de configuración del switch. Esto es un problema ya que un atacante puede recibir estos datos y detectar vulnerabilidades según la versión del switch que envió el mensaje.

Como solución podemos deshabilitar globalmente el protocolo CDP:

(config)# no cdp run

Para volver a habilitar el protocolo CDP globalmente podemos ejecutar:

(config)# cdp run

Otra opción sería deshabilitarlo en una interfaz específica:

(config)# interface <Interfaz>
(config-if)# no cdp enable

Para volver a habilitar el protocolo CDP en esa interfaz podemos ejecutar:

(config-if)# cdp enable

(Des)habilitar el protocolo LLDP

LLDP también es vulnerable a los ataques de reconocimiento como el protocolo CDP.

Como solución podemos deshabilitar globalmente el protocolo LLDP:

(config)# no lldp run

Para volver a habilitar el protocolo LLDP globalmente podemos ejecutar:

(config)# lldp run

Otra opción sería deshabilitarlo en una interfaz específica:

(config)# interface <Interfaz>
(config-if)# no lldp transmit
(config-if)# no lldp receive

Para volver a habilitar el protocolo LLDP en esa interfaz podemos ejecutar:

(config-if)# lldp transmit
(config-if)# lldp receive

Seguridad - Puertos

Deshabilitar puertos no usados

Una buena práctica es apagar todos los puertos que no se utilicen y así evitar que, si alguien intenta conectar un dispositivo no autorizado, el switch no envíe ningún tipo de tráfico a ese puerto en específico: 

(config)# interface range <Rango_de_Interfaces>
(config-if)# shutdown

En caso de que queramos conectar un dispositivo podemos encender el puerto de nuevo:

(config-if)# no shutdown

Asignar puertos no usados a la VLAN BlackHole

Es buena práctica crear una vlan llamada BlackHole y meter dentro todos los puertos que no se vayan a usar:

(config)# vlan <Número_VLAN_Sin_Usar>
(config-vlan)# name BlackHole
(config-vlan)# exit
(config)# interface range <Interfaces_Sin_Usar>
(config-if)# switchport mode access
(config-if)# switchport access vlan <Número_VLAN_BlackHole>

Seguridad - Port Security

Habilitar la seguridad de puertos

La seguridad de puertos va dirigida solo a puertos en modo acceso, y se puede activar de la siguiente forma:

(config)# interface <Interfaz>
(config-if)# switchport mode access
(config-if)# switchport port-security

Nota: Si un puerto activo está configurado con port-security y hay más de un dispositivo conectado a ese puerto, el puerto pasará al estado de error desactivado.

Seguridad de puertos > Limitar MACs

Con port-security podemos limitar la cantidad máxima de MACs que puede haber en un puerto, siendo de 1 hasta 8192:

(config)# interface <Interfaz>
(config-if)# switchport port-security maximum <Cantidad_Máxima_MACs>

Además, podemos fijar las direcciones MAC estáticas que si podrán tener acceso al puerto:

(config)# interface <Interfaz>
(config-if)# switchport port-security mac-address <Dirección_MAC>

Estas MACs que se aprendan no persisten en la configuración si se reinicia el equipo, por ello podemos definir que se guarde en NVRAM:

(config)# interface <Interfaz>
(config-if)# switchport port-security mac-address sticky

Seguridad de puertos > Caducidad de las MACs

Podemos definir que las MACs que se guardan tengan un tiempo de caducidad hasta que se eliminen de la memoria. Para las MACs dinámicas (aprendidas automáticamente) podemos configurarlo de 2 formas:

1. Caducidad absoluta - se eliminan después del tiempo de caducidad aunque estén activas:

(config)# interface <Interfaz>
(config-if)# switchport port-security aging type absolute
(config-if)# switchport port-security aging type time <Tiempo_En_Minutos>

2. Caducidad por inactividad - se eliminan solo si no transmiten tráfico durante el tiempo de caducidad:

(config)# interface <Interfaz>
(config-if)# switchport port-security aging type inactivity
(config-if)# switchport port-security aging type time <Tiempo_En_Minutos>

Para las MACs estáticas (configuradas manualmente):

(config)# interface <Interfaz>
(config-if)# switchport port-security aging static
(config-if)# switchport port-security aging type time <Tiempo_En_Minutos>

Por defecto no caducan, pero se puede habilitar el envejecimiento.

Seguridad de puertos > Modos de violación

Podemos definir que pasará si ocurre una violación de un puerto con 3 modos diferentes:

1. Que la interfaz se apague inmediatamente, generar una alerta del sistema y que se incremente el contador de violaciones:

(config)# interface <Interfaz>
(config-if)# switchport port-security violation shutdown

2. Que se descarten todos los paquetes desde y hacia esa interfaz, generar una alerta del sistema y que se incremente el contador de violaciones:

(config)# interface <Interfaz>
(config-if)# switchport port-security violation restrict

3. Que se descarten todos los paquetes desde y hacia esa interfaz sin generar ninguna alerta ni incrementar el contador de violaciones:

(config)# interface <Interfaz>
(config-if)# switchport port-security violation protect

Es el modo menos seguro de los tres.

Seguridad de puertos > Habilitar un puerto bloqueado

Si una interfaz fue bloqueada y no se configuró sticky, podemos habilitarla de nuevo:

(config)# interface <Interfaz>
(config-if)# shutdown
(config-if)# no shutdown

En caso de que sí se haya configurado sticky tendremos que quitar las direcciones MAC de la memoria:

En el siguiente comando tendremos que localizar las MACs que se guardaron en memoria (sticky):

# show run interface <Interfaz>

En el siguiente comando borraremos las líneas exactas guardadas en memoria definiendo la palabra "no" al principio:

(config-if)# no switchport port-security mac-address sticky <Dirección_MAC_A_Quitar>

Seguridad de puertos > Verificación

El siguiente comando muestra un resumen de cada interfaz con sus MACs aprendidas, el máximo y si está en modo de violación:

# show port-security

Para mostrar todas las direcciones MAC seguras que se configuran manualmente o se aprenden dinámicamente:

# show port-security address

Para verificar que una interfaz entró en modo de violación y ver su contador de violaciones podemos usar el siguiente comando:

# show port-security interface <Interfaz>

Podemos ver de una interfaz toda la configuración asignada y las MACs aprendidas en modo sticky:

# show run interface <Interfaz>

Seguridad - Ataques de VLAN

VLAN > Mitigar ataques de salto

Un atacante puede aprovechar la negociación automática de Dynamic Trunking Protocol para simular ser un switch y conseguir que el puerto negocie un enlace trunk. De esta forma podría recibir tráfico de varias VLANs.

Esto ocurre porque normalmente un equipo está conectado a un puerto en modo acceso, que solo debería tener acceso a una VLAN.

Para solucionarlo tendremos que definir los puertos del switch en modo acceso y asignarle una VLAN fija, además de deshabilitar la negociación DTP:

(config)# interface <Interfaz>
(config-if)# switchport mode access
(config-if)# switchport access vlan <Número_VLAN>
(config-if)# switchport nonegotiate

En enlaces troncales también deberíamos definir una VLAN nativa distinta a la 1, permitir solo las VLANs con tráfico y quitar la negociación de DTP:

(config)# interface <Interfaz>
(config-if)# switchport mode trunk
(config-if)# switchport trunk native vlan <Número_VLAN_Nativa>
(config-if)# switchport trunk allowed vlan <VLANs_Permitidas>
(config-if)# switchport nonegotiate

Seguridad - Ataques de ARP

DAI > Configuración

Para configurar DAI, se requiere que funcione la tabla de enlace de inspección DHCP. Por ello tendremos que habilitarlo:

(config)# ip dhcp snooping

A continuación habilitaremos en la VLAN correspondiente la inspección de DHCP y ARP:

(config)# ip dhcp snooping vlan <Número_VLAN>
(config)# ip arp inspection vlan <Número_VLAN>

Debemos configurar que los enlaces troncales sean puertos confiados/trust:

(config)# interface <Interfaz_Troncal>
(config-if)# ip dhcp snooping trust
(config-if)# ip arp inspection trust

Hay 3 tipos de validaciones de ARP:

1. Por la mac de origen - comprueba la dirección MAC de origen en el encabezado de Ethernet con la dirección MAC del remitente en el cuerpo ARP:

(config)# ip arp inspection validate src-mac

2. Por la mac de destino - comprueba la dirección MAC de destino en el encabezado de Ethernet con la dirección MAC de destino en el cuerpo ARP:

(config)# ip arp inspection validate dst-mac

3. Por la ip - comprueba el cuerpo ARP en busca de direcciones IP no válidas e inesperadas, incluidas las direcciones 0.0.0.0, 255.255.255.255 y todas las direcciones de multidifusión IP:

(config)# ip arp inspection validate ip

En las tablas aparecerá un código de letras indicando el tipo de ruta:

  • L - Indican la IP del router que se utiliza en una red conectada directamente (Local).
  • C - Identifica una red conectada directamente (Connected).
  • S - Identifica una ruta estática creada para llegar a una red específica (Static).
  • O - Identifica una red que se descubre de forma dinámica de otro router con el protocolo de routing OSPF.
  • * - La ruta es candidata para una ruta predeterminada.

Estructura de la tabla:

  • Las redes con máscara más pequeña engloban otras subredes, en las que se enseña las rutas hacia esas subredes y el tipo de ruta.
  • Las rutas con "[110/65]" significa que la ruta tiene una distancia administrativa de "110" y un costo de "65" para llegar al destino.
  • El tiempo que aparece en ciertas rutas indica el tiempo desde que se aprendió la ruta especificada.
  • La sección "via x" indica que para llegar a esa red se enviará por "x", que puede ser una IP, interfaz o ambas.

Enrutamiento estático - Rutas estáticas

IPv4 - Configurar rutas estáticas

Para definir en routers o switches de capa 3 una ruta para poder llegar a una red remota, se puede realizar de 3 maneras distintas:

1. Mediante la IP del próximo router:

(config)# ip route <IP_RedRemota> <Máscara_RedRemota> <IP_PróximoRouter>

2. Ruta Estática Conectada Directamente - define la interfaz de salida del router (solo si es de tipo Serial):

(config)# ip route <IP_RedRemota> <Máscara_RedRemota> <Interfaz_Salida>

3. Ruta estática completamente especificada - define la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ip route <IP_RedRemota> <Máscara_RedRemota> <Interfaz_Salida> <IP_PróximoRouter>

IPv6 - Configurar rutas estáticas

Para definir en routers o switches de capa 3 una ruta para poder llegar a una red remota, se puede realizar de 3 maneras distintas:

1. Mediante la IP del próximo router:

(config)# ipv6 route <IPv6_RedRemota>/<Máscara_RedRemota> <IPv6_PróximoRouter>

Importante: Si "IPv6_PróximoRouter" es de tipo link-local, como por ejemplo fe80::2, debe usarse una ruta estática completamente especificada.

2. Ruta Estática Conectada Directamente - define la interfaz de salida del router (solo si es de tipo Serial):

(config)# ipv6 route <IPv6_RedRemota>/<Máscara_RedRemota> <Interfaz_Salida>

3. Ruta estática completamente especificada - define la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ipv6 route <IPv6_RedRemota>/<Máscara_RedRemota> <Interfaz_Salida> <IPv6_Link_Local_PróximoRouter>

Enrutamiento estático - Importante

Cómo se elige la ruta a utilizar

El router selecciona la ruta que más bits de prefijo coinciden con la dirección IP de destino, es decir, la ruta más específica.

Por ejemplo, se eligirá antes una ruta estática configurada manualmente hacia una red remota sobre una ruta predeterminada.

Sumarización de subredes

Se puede ahorrar tiempo de configuración creando menos rutas. Para ello, podemos crear una dirección de red que englobe a varias subredes como por ejemplo las siguientes:

192.168.0.0/24
192.168.1.0/24

Que se pueden sumarizar a una única subred cambiando su máscara:

192.168.0.0/23

Sin sumarizar acabaremos teniendo varias rutas:

(config)# ip route 192.168.0.0 255.255.255.0 <Destino>
(config)# ip route 192.168.1.0 255.255.255.0 <Destino>

Sumarizando acabaremos teniendo una única ruta:

(config)# ip route 192.168.0.0 255.255.254.0 <Destino>

No siempre se puede sumarizar: Cuando hay subredes completamente distintas que, para sumarizar se deben englobar a otras subredes que no se desean, lo mejor es dejar las redes separadas creando así una ruta estática por cada subred.

Rutas con el mismo número de saltos

Se puede dar el caso de que por 2 interfaces diferentes se pueda llegar al mismo destino, por ello es necesario seguir unas reglas:

  1. Hay que fijarse en el coste de llegar al destino por ambas rutas. Un camino con interfaces de red Ethernet (GigabitEthernet y FastEthernet) suelen tener menor coste que una interfaz Serial.
  2. Si los caminos tiene el mismo coste de llegada, se puede crear 2 rutas estáticas sin necesidad de cambiar la distancia administrativa.

Tener 2 rutas con la misma distancia administrativa permite realizar un balanceo de carga (ECMP) al poder enviar paquetes por ambas rutas a la vez.

Enrutamiento estático - Distancia administrativa

IPv4 - Configurar rutas secundarias

La distancia administrativa permite configurar una ruta secundaria hacia una red mediante otra interfaz o IP.

Estas rutas secundarias tienen que tener un número de distancia administrativa mayor a 1, ya que la ruta primaria siempre será la 1.

1. Mediante la IP del próximo router:

(config)# ip route <IP_RedRemota> <Máscara_RedRemota> <IP_PróximoRouter> <Número_Distancia_Administrativa>

2. Ruta Estática Conectada Directamente - define la interfaz de salida del router (solo si es de tipo Serial):

(config)# ip route <IP_RedRemota> <Máscara_RedRemota> <Interfaz_Salida> <Número_Distancia_Administrativa>

3. Ruta estática completamente especificada - define la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ip route <IP_RedRemota> <Máscara_RedRemota> <Interfaz_Salida> <IP_PróximoRouter> <Número_Distancia_Administrativa>

Importante: Si creamos una misma ruta exacta pero con 2 distancias administrativas diferentes, solo estará activa la que menor distancia administrativa tenga. Para ver todas las rutas configuradas debes usar el siguiente comando: 

# show running-config | section ip route

IPv6 - Configurar rutas secundarias

La distancia administrativa permite configurar una ruta secundaria hacia una red mediante otra interfaz o IP.

Estas rutas secundarias deben tener un número de distancia administrativa mayor a 1, ya que la ruta primaria siempre será 1.

1. Mediante la IP del próximo router:

(config)# ipv6 route <IPv6_RedRemota>/<Máscara_RedRemota> <IPv6_PróximoRouter> <Número_Distancia_Administrativa>

2. Ruta Estática Conectada Directamente - define la interfaz de salida del router (solo si es de tipo Serial):

(config)# ipv6 route <IPv6_RedRemota>/<Máscara_RedRemota> <Interfaz_Salida> <Número_Distancia_Administrativa>

3. Ruta estática completamente especificada - define la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ipv6 route <IPv6_RedRemota>/<Máscara_RedRemota> <Interfaz_Salida> <IPv6_Link_Local_PróximoRouter> <Número_Distancia_Administrativa>

Importante: Si creamos una misma ruta exacta pero con 2 distancias administrativas diferentes, solo estará activa la que menor distancia administrativa tenga. Para ver todas las rutas configuradasusa el siguiente comando: 

# show running-config | section ipv6 route

Problemas de las rutas secundarias

Se suelen producir bucles de paquetes ya que los routers no son capaces de detectar que enlaces en otros routers están caídos, provocando que el router que sabe que el enlace está caído devuelva el paquete al router que no, y este lo envíe de nuevo.

La solución es usar el enrutamiento dinámico - OSPF.

Enrutamiento estático - Rutas predeterminadas

IPv4 - Configurar rutas predeterminadas

Podemos indicar a un router o switch de Capa 3 hacia donde se enviará cualquier paquete cuya IP de destino no esté en la tabla de rutas. Para ello tenemos 3 opciones:

1. Mediante la IP del próximo router:

(config)# ip route 0.0.0.0 0.0.0.0 <IP_PróximoRouter>

2. Mediante la interfaz de salida del router (solo si es de tipo Serial):

(config)# ip route 0.0.0.0 0.0.0.0 <Interfaz_Salida>

3. Definiendo la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ip route 0.0.0.0 0.0.0.0 <Interfaz_Salida> <IP_PróximoRouter>

Esto es útil en el caso de que haya routers que solo tengan una interfaz de salida, ahorrando tiempo de configurar rutas estáticas manualmente para cada subred.

IPv6 - Configurar rutas predeterminadas

Podemos indicar a un router o switch de Capa 3 hacia donde se enviará cualquier paquete cuya IP de destino no esté en la tabla de rutas. Para ello tenemos 3 opciones:

1. Mediante la IP del próximo router:

(config)# ipv6 route ::/0 <IPV6_PróximoRouter>

2. Mediante la interfaz de salida del router (solo si es de tipo Serial):

(config)# ipv6 route ::/0 <Interfaz_Salida>

3. Definiendo la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ipv6 route ::/0 <Interfaz_Salida> <IPv6_Link_Local_PróximoRouter>

Enrutamiento estático - Rutas de host

IPv4 - Configurar rutas estáticas de host

Con las rutas de host estáticas se permite definir que, para llegar a un host en concreto, hay que enviar los paquetes por cierta ruta. Se puede definir de 3 maneras:

1. Mediante la IP del próximo router:

(config)# ip route <IP_Host> 255.255.255.255 <IP_PróximoRouter>

2. Mediante la interfaz de salida del router (solo si es de tipo Serial):

(config)# ip route <IP_Host> 255.255.255.255 <Interfaz_Salida>

3. Definiendo la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ip route <IP_Host> 255.255.255.255 <Interfaz_Salida> <IP_PróximoRouter>

Nota - Estas rutas aparecerán en la tabla de rutas como - "S" (Static), además de tener una máscara /32 (IPv4).

IPv6 - Configurar rutas estáticas de host

Con las rutas de host estáticas se permite definir que, para llegar a un host en concreto, hay que enviar los paquetes por cierta ruta. Para ello tenemos 3 opciones:

1. Mediante la IP del próximo router:

(config)# ipv6 route <IPv6_Host>/128 <IP_PróximoRouter>

2. Mediante la interfaz de salida del router (solo si es de tipo Serial):

(config)# ipv6 route <IPv6_Host>/128 <Interfaz_Salida>

3. Definiendo la interfaz de salida del router (solo si es de tipo Serial) junto a la IP del próximo router:

(config)# ipv6 route <IPv6_Host>/128 <Interfaz_Salida> <IPv6_Link_Local_PróximoRouter>

Nota - Estas rutas aparecerán en la tabla de rutas como - "S" (Static), además de tener una máscara /128 (IPv6).

Enrutamiento dinámico - OSPF

Configuración básica

Para configurar OSPF, debemos habilitarlo explícitamente mediante el siguiente comando, especificando un número de proceso entre 1 - 65535:

(config)# router ospf <Número_Proceso>
(config-router)# ...

Es recomendable usar el mismo número de proceso en cada router de la red que habiliten OSPF.

Asignar un Router ID

Cada router que use OSPF debe tener un identificador único en formato de dirección IPv4. No importa si esta dirección IPv4 está en uso, ya que no se utiliza para la comunicación de datos:

(config)# router ospf <Número_Proceso>
(config-router)# router-id <Dirección_IPv4_Única>

Es recomendable que esta dirección IPv4 coincida con la configurada en una interfaz de loopback.

Para verificar el Router ID del dispositivo actual se puede usar el siguiente comando:

# show ip protocols | include Router ID

Recargar la configuración

Un router no permitirá un cambio de Router ID a no ser que se recargue su configuración previamente:

# clear ip ospf process

Pedirá una confirmación, en la que tendremos que poner y.

Definir redes

Para definir las interfaces que se compartirán por OSPF tenemos 3 maneras diferentes de configurarlo:

1. Por dirección de red:

Es importante definir la máscara de la dirección de red en formato Wildcard. Este formato resta 255.255.255.255 a la máscara de red en formato punteado.

# show ip route connected
(config)# router ospf <Número_Proceso>
(config-router)# network <Dirección_Red> <Máscara_Wildcard> area <ID_Área>

2. Por la dirección IP de la interfaz:

Por cada interfaz de red ejecutaremos el siguiente comando especificando su dirección IP con máscara 0.0.0.0.

# show ip interface brief | include up
(config)# router ospf <Número_Proceso>
(config-router)# network <Dirección_IP> 0.0.0.0 area <ID_Área>

3. Por interfaz:

Por cada interfaz de red ejecutaremos el siguiente comando especificando el número de proceso OSPF y el área en el que se sitúa.

# show ip interface brief | include up
(config)# interface <Interfaz>
(config-router)# ip ospf <Número_Proceso> area <ID_Área>

Eliminar redes

Para eliminar una red configurada en OSPF simplemente pondremos la palabra "no" al principio del comando:

(config)# router ospf <Número_Proceso>
(config-router)# no network <Dirección_Red> <Máscara_Wildcard> area <ID_Área>

Para ver las redes configuradas podemos usar el siguiente comando:

# show running-config | include network

Interfaz pasiva

Configurar una interfaz como pasiva no reenvía e ignora mensajes de tipo OSPF:

(config)# router ospf <Número_Proceso>
(config-router)# passive-interface <Interfaz>

Para quitar una interfaz del modo pasivo hay que añadir la palabra "no" al principio del comando:

(config)# router ospf <Número_Proceso>
(config-router)# no passive-interface <Interfaz>

Tipo de interfaz punto a punto

Si el tipo de red se detecta como BROADCAST, pero en realidad se trata de un enlace punto a punto, se puede modificar manualmente con:

(config)# interface <Interfaz>
(config-if)# ip ospf network point-to-point
# show ip ospf interface <Interfaz>

Al configurar la interfaz como point-to-point:

  • Se deshabilita la elección de DR y BDR.
  • Se establece una adyacencia directa entre los dos routers.
  • Se simplifica el proceso de OSPF en ese enlace.

Configuración de la prioridad de una interfaz

Asignar una prioridad a una interfaz permite asignar manualmente qué routers será el DR o BDR según el que tenga una prioridad más alta.

Las prioridades van de 1 - 255, aunque la prioridad 0 se usa para indicar que no debe ser nunca DR ni BDR:

(config)# interface <Interfaz>
(config-if)# ip ospf priority <Prioridad>

Después de cambiar la prioridad se debe reiniciar la configuración (puede interrumpir temporalmente el tráfico):

# clear ip ospf process

Configuración del ancho de banda

En cada router del dominio OSPF se debe configurar el coste de ancho de banda si hay enlaces más rápidos que Fast Ethernet, que todos los costes por defecto son 100 Mbps, y esto puede provocar que se prefieran otros enlaces en vez de los más rápidos, como Gigabit.

Para cambiar este coste se debe usar el siguiente comando indicando el coste en Megabits por segundo, como por ejemplo "1000" si el ancho de banda es de 1 Gbps:

(config)# router ospf <Número_Proceso>
(config-router)# auto-cost reference-bandwidth <Velocidad_En_Mbps>

Con el siguiente comando se puede ver el coste de ancho de banda representado en Gbps:

# show ip ospf interface <Interfaz>

La suma de todos los costes para llegar a una red en específico se puede ver directamente con el siguiente comando:

# show ip route | include <Dirección_Red>

Enseñará en la primera línea 2 números entre corchetes, como por ejemplo [110/11]. El "11" indica ese coste total.

Nota: El coste se calcula dividiendo 100 Mbps entre el ancho de banda configurado en la interfaz en Mbps.

Configuración del coste

La configuración manual del coste en un enlace permite preferir una ruta hacia un destino sobre otra ruta (cuanto más bajo sea, mejor). Configuración:

(config)# interface <Interfaz>
(config-if)# ip ospf cost <Coste>

Para verificar los cambios se puede localizar la ruta con el siguiente comando y verificar que el segundo número entre corchetes sea la suma de todos los costes configurados:

# show ip route | include <Dirección_Red>

Intervalo de paquetes: Hello y Dead

Por cada interfaz hay un intervalo de envío de paquetes OSPF de tipo Hello y Dead. Estos intervalos se pueden consultar con el siguiente comando:

# show ip ospf interface <Interfaz>

Aparecerá en la sección Timer intervals configured.

También se puede mirar en la tabla de vecinos el tiempo actual de Dead por cada vecino:

# show ip ospf neighbor

Para modificar el intervalo de paquetes Hello, hay que usar el siguiente comando dentro de la interfaz deseada:

(config)# interface <Interfaz>
(config-if)# ip ospf hello-interval <Tiempo_Segundos>

El intervalo de paquetes Dead también se configuran dentro de la interfaz:

(config-if)# ip ospf dead-interval <Tiempo_Segundos>

⚠️ Importante: Los routers vecinos deben tener el mismo intervalo de paquetes Hello o se perderá la adyacencia.


Para restablecer el intervalo de paquetes Hello, hay que utilizar el siguiente comando dentro de la interfaz:

(config)# interface <Interfaz>
(config-if)# no ip ospf hello-interval

El intervalo de paquetes Dead también se reestablece dentro de la interfaz:

(config-if)# no ip ospf dead-interval

Propagación de la ruta predeterminada

El router que tenga acceso a Internet tendrá que crear una ruta estática predeterminada y propagarla por OSPF. Configuración:

1. Se debe crear una ruta predeterminada estática:

(config)# ip route 0.0.0.0 0.0.0.0 <Destino>

2. Se debe permitir la propagación por OSPF:

(config)# router ospf <Número_Proceso>
(config-router)# default-information originate

⚠️ Importante: Suele ser necesario reiniciar Packet Tracer para que funcione correctamente.

Una vez configurado, se debe poder ver en cada router una ruta predeterminada aprendida por OSPF con destino el router que lo propagó:

# show ip route ospf

Diagnóstico de fallos

1. Configuración de OSPF:

El primer paso es verificar que el siguiente comando devuelva algo, indicando que OSPF está configurado: 

# show ip protocols

2. Interfaces pasivas:

En el comando del paso 1, se puede comprobar que cada interfaz que tenga conexión a otro router no esté en modo pasivo. Las únicas interfaces que deben estar en modo pasivo son aquellas que no tengan conexión a ningún router.

3. Ruta a Internet:

En el comando del paso 1, hay que verificar que el router con acceso a Internet no esté anunciando en OSPF la red externa (ISP). En su lugar, se recomienda usar una ruta por defecto y propagarla por OSPF.

4. Fuentes de información:

En el comando del paso 1, se puede comprobar que en una misma área estén todos los routers, incluido el mismo router, que ejecute el comando como fuentes de información.

5. Router ID:

Es necesario verificar que cada router tenga un Router ID y que sea único: 

# show running-config | include router-id

6. Áreas:

Hay que verificar que se hayan especificado las áreas correctas: 

# show running-config | include area

7. Comprobación de vecinos:

Es importante comprobar que cada router detecte correctamente a sus vecinos: 

# show ip ospf neighbor

El estado debe ser FULL. Si es INIT o EXSTART puede deberse a problemas de MTU o parámetros.

8. Interfaces configuradas:

Con el siguiente comando se puede verificar que todas las interfaces necesarias estén en OSPF: 

# show ip ospf interface brief

9. Intervalos configurados:

Como se explicó en Intervalo de paquetes: Hello y Dead, si los timers son diferentes entre vecinos se perderá la adyacencia: 

# show ip ospf interface

10. Tipo de red:

En el comando del paso 9, se puede comprobar que el tipo de red sea BROADCAST o POINT-TO-POINT, según el caso que sea, además de que el coste y estado sean los correctos.

11. Cantidad de rutas aprendidas:

Es muy importante comprobar que las rutas que se aprendan (P2P y conectadas a otros routers) sean detectadas correctamente por cada router. Puede haber una ruta duplicada haciendo que se detecte solo 1 en vez de las 2 que deberia haber. Comando de comprobación: 

# show ip route

Enrutamiento dinámico - OSPF Multiárea

Teoría

OSPF usa áreas para facilitar la administración y optimizar el uso de recursos y el tráfico.

Un área es un grupo lógico de redes y routers conectados, donde todos los routers comparten la misma información de topología y no conocen lo que ocurre en otras áreas.

La comunicación entre áreas se realiza mediante routers de borde de área (ABR), que conectan varias áreas, incluyendo siempre el área 0 (BackBone), que es la principal en el diseño de la red.

Fuente de información - Hay que tener en cuenta que solo los routers de una misma área son fuentes de información de esa área.

Desventaja de OSPF Multiárea - El camino hacia el destino no siempre es el óptimo.

Configuración

1. En cada router de la red hay que configurar el mismo ID de proceso junto a un Router ID único:

(config)# router ospf <Número_Proceso>
(config-router)# router-id <Dirección_IPv4_Única>

2. En los routers de área hay que configurar las redes del área en el que se sitúan:

(config)# interface <Interfaz>
(config-if)# ip ospf <Número_Proceso> area <ID_Área>
(config-if)# ...

3. En el router de borde de área (ABR) hay que configurar sus áreas conectadas además de la área 0:

(config)# interface <Interfaz>
(config-if)# ip ospf <Número_Proceso> area <ID_Área>
(config-if)# ...

4. La sumarización se utiliza para que un área publique una sola ruta consolidada hacia las demás áreas, reduciendo el tamaño de las tablas de enrutamiento:

(config)# router ospf <Número_Proceso>
(config-router)# area <ID_Área_Origen> range <IP_Red_Sumarizada> <Máscara_Subred>

ID_Área_Origen: Es el ID del área donde residen originalmente las subredes antes de ser enviadas al resto de la red.
IP_Red_Sumarizada: Es la dirección de red del conjunto de IPs que engloban a toda o gran parte del área.

5. Por último, hay que realizar una verificación de la conectividad con sus routers vecinos y las rutas aprendidas por OSPF.

ACL

Donde ubicar las ACL

Las ACL estándar deben aplicarse lo más cerca posible del destino del tráfico.

Las ACL extendidas deben ubicarse lo más cerca posible del origen del tráfico que se desea filtrar.

En qué orden se deben crear las ACL

El orden de las reglas dentro de una ACL es fundamental, ya que el router evalúa las entradas de forma secuencial, de arriba hacia abajo.

En el momento en que un paquete coincide con una regla, se aplica esa acción (permit o deny) y no se continúan evaluando las reglas siguientes.

Por este motivo:

  • Las reglas más restrictivas deben colocarse al inicio de la ACL.
  • Las reglas más generales o menos restrictivas deben ubicarse al final.

A tener en cuenta

Por defecto, todo el tráfico está denegado implícitamente. Esto significa que, si no se especifican reglas de permit, todo será bloqueado automáticamente.

Por lo tanto, una ACL que solo contenga reglas de deny no tendrá efecto práctico si no incluye también reglas de permit, ya que el comportamiento por defecto ya es denegar todo.

ACL Estándar

Sintaxis

(config)# access-list <Número_ACL> <permit / deny> <A_Quien> <log>

Número_ACL: Es un número identificativo para la ACL, que van del 1 a 99, o 1300 a 1999.
permit / deny: Se define implícitamente si se permite o deniega el tráfico.
A_Quien: Se define la dirección de red y su máscara Wildcard o la IP de un host al que se permite o deniega el tráfico.
log: Si se pone la palabra "log" al final del comando, el sistema avisará cuando esta regla se cumple y aplica (opcional).

ACL asignada a una dirección de red

Para definir a quien se le aplica la ACL se debe usar una IP con una máscara WildCard, que se calcula restando 255.255.255.255 a la máscara punteada. Un ejemplo sería la máscara /24 (255.255.255.0) que, restada a 255.255.255.255, daría como resultado 0.0.0.255. 

(config)# access-list <Número_ACL> <permit / deny> <Dirección_Red> <Máscara_Wildcard>

Mediante direcciones de red podremos sumarizar para evitar crear muchas reglas, como por ejemplo:

Con una máscara /23 podemos resumir dos redes /24 (192.168.0.0 y 192.168.1.0) en una sola regla, evitando definirlas manualmente:

(config)# access-list 10 permit 192.168.0.0 0.0.1.255

Sin sumarizar tendríamos 2 reglas a configurar:

(config)# access-list 10 permit 192.168.0.0 0.0.0.255
(config)# access-list 10 permit 192.168.1.0 0.0.0.255

ACL asignada a un host

Las siguientes 2 formas cumplen con la función de permitir o denegar el tráfico de una única IP - IP_Host: 

(config)# access-list <Número_ACL> <permit / deny> host <IP_Host>
(config)# access-list <Número_ACL> <permit / deny> <IP_Host> 0.0.0.0

ACL asignada a todas las IPs

Las siguientes 2 formas cumplen con la función de permitir o denegar el tráfico a todas las IPs que existen: 

(config)# access-list <Número_ACL> <permit / deny> any
(config)# access-list <Número_ACL> <permit / deny> 0.0.0.0 255.255.255.255

ACL Extendida

Sintaxis

(config)# access-list <Número_ACL> <permit / deny / remark [Texto]> <Protocolo> <Origen> [Puerto_Origen] <Destino> [Puerto_Destino] [established] [log]

Número_ACL: Es un número identificativo para la ACL, que van del 100 a 199, o 2000 a 2699.
permit / deny: Se define implícitamente si se permite o deniega el tráfico.
Protocolo: Se define qué a qué tipo de tráfico se refiere, como "tcp", "udp", "icmp", "ospf", "ip" (cualquier protocolo de Internet), etc.

Origen: Es la IP y máscara WildCard de los hosts de origen.
Puerto_Origen: Es el puerto de host de origen a comparar (opcional).

Destino: Es la IP y máscara WildCard de los hosts de destino.
Puerto_Destino: Es el puerto de host de destino a comparar (opcional).

established: Si se pone la palabra "established", solo se comparará si la conexión es TCP y se ha establecido la conexión (opcional).
log: Si se pone la palabra "log" al final del comando, el sistema avisará cuando esta regla se cumple y aplica (opcional).

Para definir el puerto se puede hacer de distintas maneras:

  • eq (Puerto) - Que el puerto sea igual al puerto especificado.
  • lt (Puerto) - Que el puerto sea menor al puerto especificado.
  • gt (Puerto) - Que el puerto sea mayor al puerto especificado.
  • neq (Puerto) - Que el puerto no sea igual al puerto especificado.
  • El puerto puede ser un número o un nombre que Cisco defina, como por ejemplo "www" para el puerto "80" o "ssh" para el puerto "22".

Ejemplos

Tráfico WEB:

Permitir solo el tráfico HTTP/TCP y HTTPS/TCP de la red 192.168.10.0/24 a cualquier destino:

(config)# access-list 100 permit tcp 192.168.10.0 0.0.0.255 any eq 80
(config)# access-list 100 permit tcp 192.168.10.0 0.0.0.255 any eq 443

Esta ACL "100" debería asignarse a la entrada de tráfico de la interfaz donde el router tenga conectado la red 192.168.10.0/24.

Opcionalmente también se puede limitar las respuestas de entrada a la red desde Internet para que solo se permita el tráfico de las conexiones TCP que se hayan iniciado en la red local, evitando así intentos de conexiones externas a equipos de la red: 

(config)# access-list 101 permit tcp any 192.168.10.0 0.0.0.255 established

Esta ACL "101" debe aplicarse a los paquetes salientes de la interfaz conectada a la red 192.168.10.0/24 (out).

Tráfico SSH:

Denegar el tráfico SSH/TCP de cualquier IP a la red local 192.168.10.0/24:

(config)# access-list 102 deny tcp any 192.168.10.0 0.0.0.255 eq 22
(config)# access-list 102 permit ip any any

Esta ACL "102" debería asignarse a la entrada de tráfico de la interfaz donde el router tenga conectado la red 192.168.10.0/24.
Sin la segunda línea no se permitiría ningún tipo de tráfico ya que todo lo que no se permita se deniega por defecto.

Más funciones de las ACL

Crear ACLs extendidas con un nombre

Para crear ACLs se puede hacer con un número identificativo (como se vió hasta ahora) o mediante un nombre identificativo y único. De esta forma no hace falta poner "access-list Número" al principio de cada regla permit / deny. La única regla es que no debe haber espacios en el nombre.

Si es una ACL estándar:

(config)# ip access-list standard <Nombre_ACL>
(config-std-nacl)# <permit / deny> <A_Quien>
(config-std-nacl)# ...

Si es una ACL extendida:

(config)# ip access-list extended <Nombre_ACL>
(config-ext-nacl)# <permit / deny> <Protocolo> <Origen> [Puerto_Origen] <Destino> [Puerto_Destino] [established] [log]
(config-ext-nacl)# ...

Asignar un comentario a una ACL extendida

Un comentario permite asignar una descripción breve de hasta 100 carácteres a cada ACL, permitiendo identificarlas fácilmente. Se puede crear comentarios en una lista identificada por un número: 

(config)# access-list <Número_ACL> remark <Texto>

También se puede poner comentarios si la ACL se creó por un nombre:

Si es una ACL estándar:

(config)# ip access-list standard <Nombre_ACL>
(config-std-nacl)# remark <Texto>

En una ACL extendida se hace igual pero cambiando "standard" por "extended".

Como borrar una regla del ACL

Primero debes que número de secuencia tiene la regla en la lista:

# show access-lists

Métete a la ACL para eliminar la regla con el número de secuencia del paso anterior y luego crea la regla de nuevo pero indicando el número de secuencia:

(config)# ip access-list standard <Número_ACL / Nombre_ACL>
(config-std-nacl)# no <Número_Secuencia_Regla>
(config-std-nacl)# <Número_Secuencia_Regla> <Pon_La_Regla_Aquí>

En una ACL extendida se hace igual pero cambiando "standard" por "extended".

Como añadir una regla en una posición especifica de la ACL

Primero debes mirar entre qué reglas se va a situar la nueva regla:

# show access-lists

Ahora escoge un número de secuencia libre entre esas 2 reglas. Por ejemplo, para crear una regla entre las reglas con número de secuencia 10 y 20, puedes crear la nueva regla con un número de secuencia 15.

Métete a la ACL a editar para ingresar la nueva regla con el número de secuencia escogido en el paso anterior:

(config)# ip access-list standard <Número_ACL / Nombre_ACL>
(config-std-nacl)# <Número_Secuencia_Intermedio_Regla> <Crea_La_Regla_Aquí>

En una ACL extendida se hace igual pero cambiando "standard" por "extended".

Asignar una ACL

Asignar una ACL a una interfaz

Primero hay que meterse en la interfaz a la que se le vaya a aplicar la ACL:

(config)# interface <Interfaz>

Y luego hay que ejecutar el siguiente comando especificando la ACL mediante su número o nombre (según como se haya creado), seguido de que si se aplica a paquetes entrantes (in) o salientes (out): 

(config-if)# ip access-group <Número_ACL / Nombre_ACL> <in / out>

Ejemplo: Se permite a la red acceder a sitios web y solo se permiten conexiones TCP a la red si están establecidas:

(config)# ip access-list extended SURFING
(config-ext-nacl)# permit tcp 192.168.10.0 0.0.0.255 any eq 80
(config-ext-nacl)# permit tcp 192.168.10.0 0.0.0.255 any eq 443
(config-ext-nacl)# exit
(config)#
(config)# ip access-list extended BROWSING
(config-ext-nacl)# permit tcp any 192.168.10.0 0.0.0.255 established
(config-ext-nacl)# exit
(config)#
(config)# interface g0/0/0
(config-if)# ip access-group SURFING in
(config-if)# ip access-group BROWSING out
(config-if)# end

Asignar una ACL a las líneas VTY

Para permitir a un unos equipos específicos el acceso remoto VTY de un router tendremos que crear primero una ACL:

(config)# ip access-list standard <Nombre_ACL>
(config-std-nacl)# permit host <IP_Host>
(config-std-nacl)# deny any

Luego tendremos que meternos a las conexiones VTY y aplicar la ACL:

(config)# line vty 0 15
(config-line)# access-class <Nombre_ACL> in

Si el acceso remoto no está configurado todavía, se recomienda seguir la guía: Configuración de un servidor SSH.

Verificación de ACL

Comandos de verificación

Con el siguiente comando se pueden ver todas las ACL creadas junto a sus reglas y el número de secuencia correspondiente a cada regla:

# show access-lists

Esto también muestra un contador por cada regla que indica cuántos paquetes se denegaron o permitieron. Para reiniciar estos contadores:

# clear access-list counters <Número_ACL / Nombre_ACL>

También se puede ver las ACL y sus reglas en la configuración del equipo, pero no se enseñan los números de secuencia:

# show run | section access-list

Para verificar en una interfaz que ACL tiene aplicadas:

# show ip interface <Interfaz> | include access list

NAT estática

Configuración con traducción de IPs

La NAT estática permite establecer una traducción fija entre una IP privada y una IP pública, de modo que el host interno puede ser accesible desde el exterior y, al mismo tiempo, utiliza siempre esa misma IP pública para comunicarse hacia Internet.

Con la NAT estática se requiere que haya suficientes direcciones públicas disponibles para satisfacer la cantidad total de sesiones de usuario simultáneas.


1. Se vincula una IP local interna (privada) con una IP global interna (pública):

(config)# ip nat inside source static <IP_Local_Interna> <IP_Global_Interna>

2. Ahora hay que definir que interfaces actuarán como interna y externa:

Interna - Son aquellas interfaces conectadas a la red privada (LAN):

(config)# interface <Interfaz_Interna>
(config-if)# ip nat inside

Externa - Son aquellas interfaces conectadas a la red pública (WAN):

(config)# interface <Interfaz_Externa>
(config-if)# ip nat outside

A estas interfaces también hay que configurarles una IP y definir que estén activas.

Configuración con traducción de IPs y puertos

A continuación se muestra la sintaxis para traducir la IP local interna (privada) a la IP global interna (pública) mediante puertos:

(config)# ip nat inside source static <Protocolo> <IP_Local_Interna> <Puerto_Local> <IP_global_interna> <Puerto_Global>

Protocolo: Se define qué a qué tipo de tráfico se refiere, como "tcp", "udp", "icmp", "ospf", "ip" (cualquier protocolo de Internet), etc.

IP_Local_Interna: Es la IP privada del host (LAN).
Puerto_Local: Es el puerto del servicio en el host interno (LAN).

IP_Global_Interna: Es la IP pública del router (WAN).
Puerto_Global: Es el puerto del router expuesto hacia Internet (WAN).

Un ejemplo sería el siguiente:

(config)# ip nat inside source static tcp 192.168.0.2 80 182.204.54.130 80

El router redirige todo el tráfico:

  • que llegue a 182.204.54.130:80 (IP pública)
  • hacia 192.168.0.2:80 (servidor interno)

NAT dinámica

Configuración con una pool de direcciones

La NAT dinámica permite asignar de forma automática una dirección IP pública a un host interno a partir de un conjunto de direcciones disponibles (pool), de modo que cada host interno recibe una IP pública del pool mientras haya disponibles.

Nota: Cada IP pública del pool solo puede ser utilizada por un host a la vez.


1. Crear una ACL estándar, definiendo solo las redes que van a ser traducidas:

(config)# access-list <Número_ACL> permit <A_Quien>

2. Definir el conjunto de direcciones disponibles (pool) para la traducción:

(config)# ip nat pool <Nombre_Pool> <Primera_IP> <Ultima_IP> netmask <Máscara_Red>

Nombre_Pool: Nombre identificativo del pool.
Primera_IP: Primera IP disponible del pool.
Ultima_IP: Última IP disponible del pool.
Máscara_Red: máscara en formato punteado.

3. Enlazar la ACL del paso 2 con la pool creada en el paso 1:

(config)# ip nat inside source list <Número_ACL> pool <Nombre_Pool>

4. Ahora hay que definir que interfaces actuarán como interna y externa:

Interna - Son aquellas interfaces conectadas a la red privada (LAN):

(config)# interface <Interfaz_Interna>
(config-if)# ip nat inside

Externa - Son aquellas interfaces conectadas a la red pública (WAN):

(config)# interface <Interfaz_Externa>
(config-if)# ip nat outside

Nota: Las traducciones dinámicas se crean cuando hay tráfico y se eliminan tras un tiempo de inactividad.

Borrar traducciones dinámicas activas

Para borrar las entradas dinámicas antes de que expire el tiempo de espera:

# clear ip nat translation *

Para borrar solo una entrada dinámica en específico:

# clear ip nat translation inside <Global_IP> <Local_IP> [outside <Global_IP> <Local_IP>]

Para borrar solo una entrada dinámica extendida en específico:

# clear ip nat translation {tcp | udp} inside <Global_IP> <Global_Port> <Local_IP> <Local_Port> [outside <Global_IP> <Global_Port> <Local_IP> <Local_Port>]

PAT

Configuración para usar una única IPv4

PAT permite que múltiples hosts internos compartan una única dirección IP pública, diferenciando cada conexión mediante el uso de números de puerto. De esta forma, varios dispositivos de la red privada pueden acceder simultáneamente a redes externas, ya que el router traduce no solo la dirección IP, sino también el puerto de origen, asignando combinaciones únicas para cada sesión.

A diferencia de NAT dinámica, varias direcciones IP privadas pueden compartir una misma IP pública simultáneamente gracias a la traducción de puertos.


1. Crear una ACL estándar, definiendo solo las redes que van a ser traducidas:

(config)# access-list <Número_ACL> permit <A_Quien>

2. Vincular la ACL con la interfaz externa conectada a la red pública (WAN):

(config)# ip nat inside source list <Número_ACL> interface <Interfaz> overload

3. Ahora hay que definir que interfaces actuarán como interna y externa:

Interna - Son aquellas interfaces conectadas a la red privada (LAN):

(config)# interface <Interfaz_Interna>
(config-if)# ip nat inside

Externa - Son aquellas interfaces conectadas a la red pública (WAN):

(config)# interface <Interfaz_Externa>
(config-if)# ip nat outside

1. Crear una ACL estándar, definiendo solo las redes que van a ser traducidas:

2. Definir el conjunto de direcciones disponibles (pool) para la traducción:

Ultima_IP: Última IP disponible del pool.
Máscara_Red: máscara en formato punteado.

3. Vincular la ACL del paso 1 con la pool creada en el paso 2:

4. Ahora hay que definir que interfaces actuarán como interna y externa:

(config)# interface <Interfaz_Interna>

Externa - Son aquellas interfaces conectadas a la red pública (WAN):

NAT - Verificación

Verificación de la configuración NAT

Este comando permite ver las traducciones NAT activas:

# show ip nat translations

Las traducciones NAT estáticas siempre aparecen en la tabla, aunque no estén activas en ese momento.
Cuando una traducción está activa, se muestra en una entrada adicional donde se especifica el protocolo de comunicación y las direcciones IP correspondientes a Outside local y Outside global.

Añadiendo la palabra verbose al comando anterior se puede ver información adicional sobre cada traducción, incluido cuánto tiempo hace que se creó y usó la entrada:

# show ip nat translations verbose

El tiempo de expiración depende del tipo de tráfico, aunque por defecto puede ser de hasta 24 horas. Para cambiarlo usa el siguiente comando: 

(config)# ip nat translation timeout <Tiempo_En_Segundos>

Este comando muestra información sobre el número total de traducciones activas, las pools configuradas y todas las interfaces configuradas como internas y externas:

# show ip nat statistics

También hay una fila de estadísticas "Hits: 0 Misses: 0", donde:

  • Hits - número de paquetes que han coincidido con una traducción NAT existente.
  • Misses - número de paquetes que no encontraron una traducción existente y requirieron la creación de una nueva entrada NAT (o no pudieron ser traducidos).

Para reiniciar estas estadísticas puedes usar:

# clear ip nat statistics

Alternativamente, para ver la pool configurada se puede ver directamente desde la configuración del dispositivo:

# show running-config | include nat

Cómo identificar una interfaz interna o externa

Con el siguiente comando podremos ver todas las interfaces activas con sus IPs configuradas:

# show ip interface brief | include up
  • Una interfaz interna (inside) suele estar conectada a la red local (LAN) y normalmente tiene una dirección IP privada.
  • Una interfaz externa (outside) está conectada a la red pública o a otra red externa (WAN), y suele tener una dirección IP pública, aunque no siempre es así.

Hay que tener en cuenta que NAT o PAT se configura únicamente en el router que separe la red local (LAN) con una red externa (WAN).

Compartir rutas

Configuración de BGP

Es necesario entender que BGP se debería usar solo cuando un sistema autónomo (AS) tiene conexión con mínimo otros 2 sistemas autónomos.

1. Primero hay que habilitar BGP especificando el Número de Sistema Autónomo (ASN) correspondiente a la entidad:

(config)# router bgp <Número_ASN>

Tipos de números ASN:

  • Privado - Cualquier número entre 64512 y 65534.
  • Público - Cualquier número asignado directamente por la IANA.

2. Ahora hay que configurar cada entidad vecina indicando su IP y el número ASN al que pertenece:

(config-router)# neighbor <IP_Vecino> remote-as <Número_ASN_Vecino>

3. Una vez configurados los pasos anteriores en todas las entidades, debería aparecer un mensaje similar al siguiente:

%BGP-5-ADJCHANGE: neighbor <IP_Vecino> Up

4. En cada entidad se deben definir las redes que se desean publicar a otras entidades:

(config-router)# network <Dirección_Red> mask <Máscara_Red>

Que redes publicar:

  • Redes propias de la entidad.
  • Redes que deban ser alcanzables desde otros sistemas autónomos.
  • Redes internas que se quieran hacer públicas mediante BGP.

Que redes no se recomienda publicar:

  • Redes de gestión interna.
  • Redes no utilizadas.
  • Prefijos demasiado específicos innecesariamente.

Importante - Las redes solo se anunciarán si existe previamente una ruta hacia ellas en la tabla de enrutamiento. En caso necesario, puede crearse una ruta estática hacia Null0 únicamente para que la red exista en la tabla de rutas: 

(config-router)# ip route <Dirección_Red> <Máscara_Red> Null0

Esta ruta no envía tráfico a ningún destino real; simplemente permite anunciar la red mediante BGP.

Verificación de BGP

Para comprobar que BGP funciona se puede ver que las rutas a redes publicadas por los sistemas autónomos se vean desde el resto de AS:

# show ip route

Las rutas creadas por BGP se indican mediante la letra "B".
